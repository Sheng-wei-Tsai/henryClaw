# Repository Memory

## Stable Context
- 🦞 這個 repo 目前可確認的長期脈絡非常薄，**沒有足夠證據支持任何穩定的產品方向、技術架構、開發流程或生活節奏規律**。龍蝦群目前只能誠實記住：可驗證內容很少。
- 🦞 近一段時間內唯一反覆被提到的 issue 是 **#1**，而且它已在 **2026-05-15** 關閉。從多日快照一致判讀來看，**#1 應視為沒有形成實質需求的占位／測試性條目**，不是可延續的工作主線。
- 🦞 這個 repo 的記憶整理目前偏向**清理追蹤入口與維持記錄可信度**，而不是回顧大量實作歷程。換句話說，龍蝦們在這裡更常做的是辨識「哪些其實不是任務」，而不是整理已完成的功能。
- 🦞 若未來出現真正要承接的工作，**應以新的、具備目標與驗收條件的 issue 或補齊背景的紀錄為準**；現有資料不足以把舊的空白條目當作可靠起點。

## Recent Themes
- 🦞 **議題衛生整理**：多日快照都在重複同一件事——確認 #1 沒有實質內容，並把它從「可能待辦」降級為「已結束的空白紀錄」。
- 🦞 **避免虛構脈絡**：龍蝦群反覆採取保守策略，不從空白 issue 推導不存在的需求、不把缺乏證據的活動寫成進展。
- 🦞 **上下文匯流不足**：多份 daily snapshot 都提到，若 Telegram 或其他外部討論沒有匯入，repo 內就無法還原主人真正的工作背景。這是近期反覆浮現的記憶缺口。
- 🦞 **目前看不出穩定工程主題**：最近的資料沒有顯示功能開發、除錯、規格討論、部署、文件補強等持續主線；這不是代表一定沒做事，而是**現有記錄無法證明**。

## Constraints
- 🦞 **資料來源受限**：目前主要依賴 GitHub issue 與其可見內容；若外部對話未匯入，龍蝦們就無法可靠重建上下文。
- 🦞 **不可腦補需求**：在沒有目標、背景、討論或驗收條件時，不應把 issue 擴寫成任務，這是目前最重要的記憶準確性原則。
- 🦞 **#1 不應再被當作主要追蹤入口**：除非後續有人補齊完整背景，否則它只是一張已關閉的占位卡，不適合作為續做依據。
- 🦞 **整體資訊不足且存在不確定性**：目前無法確認主人是否有其他未被記錄的工作流、生活事件或技術決策正在進行；缺資料本身就是限制。

## Open Loops
- 🦞 **#1 是否其實對應某段外部討論或真實需求？** 目前無法確認；若有 Telegram 或其他來源的脈絡，需補匯入後再判讀。
- 🦞 **若要重啟 #1 或其相關想法**，需要先補上：
  - 明確目標
  - 背景脈絡
  - 預期產出
  - 驗收條件
- 🦞 **repo 的真實工作主線仍不清楚**：現有 snapshots 無法辨識近期是否有其他工程主題正在推進、卡住或已完成。
- 🦞 **記憶系統的上下文完整性仍待改善**：若主人希望 daily memory 能反映更完整的生活／工作流，需要讓外部討論與 repo 內追蹤資料更穩定對齊。
