# Repository Memory

## Stable Context
- 龍蝦群目前能穩定確認的 repo 脈絡非常少；近一段時間的 issue 記錄幾乎沒有形成可延續的產品、功能、除錯或文件主線。
- 唯一反覆出現且可定調的歷史事件是：`#1` 已被判定為**沒有實質內容的占位 issue**，且已關閉，不應再視為有效待辦或工作依據。
- 這個 repo 目前更像是在維持**待辦可信度與記憶整潔**，而不是承載大量活躍開發脈絡。
- 對龍蝦群來說，issue 是長期記憶的重要入口；若沒有明確目標、背景、討論或驗收條件，就不應自動推論成真實需求。

## Recent Themes
- 近期反覆出現的主題不是功能推進，而是**清理空白任務、降低 backlog 噪音**。
- 多日快照都在收斂同一件事：`#1` 沒有需求描述、沒有討論、沒有可執行上下文，因此被視為占位卡並結案。
- 近期整理視窗內，**沒有可辨識的跨 issue 主題**；到 2026-06-14 為止，甚至沒有新的 issue 活動可供整理。
- 龍蝦群觀察到一個流程訊號：主人或協作者傾向在資料不足時先做**去噪與收斂**，而不是硬把空白紀錄延伸成假進度。
- 不確定性很高：目前看不出這代表 repo 本身暫時低活躍，還是只是工作沒有被 issue 化。

## Constraints
- **資料來源限制明顯**：近 30 天可用 issue 資料極少，且內容稀薄，無法可靠還原主人的完整工作／生活節奏。
- **不可腦補需求**：對於像 `#1` 這種缺少目標、背景、討論與驗收條件的紀錄，龍蝦群應避免自行延伸推論。
- **舊 issue 不應承接新需求**：若原卡片是空白占位且已關閉，後續若要重啟方向，應以新 issue 重新建立上下文。
- **記憶品質依賴 issue 品質**：若任務沒有被清楚記錄，daily snapshots 只能誠實反映「資料不足」，不能保證捕捉到真實進展。
- 目前**沒有足夠證據**支持任何穩定的技術棧偏好、協作流程細節、生活習慣或固定優先序；這些都應保持未定。

## Open Loops
- 最大的未解點是：`#1` 當初是否其實對應某個真實想法，只是沒有寫下來。若主人還記得，需**另開新 issue** 並補上：
  - 任務目標
  - 背景脈絡
  - 預期產出或驗收條件
  - 若有外部對話來源（如聊天紀錄），附上摘要或連結
- 目前無法判斷是否存在**未被 issue 化的實際工作**；這是記憶缺口，不是已確認的「沒有工作」。
- 等待後續新的 issue 活動，才能判斷這個 repo 是否會形成穩定主線，或持續維持低訊號狀態。
