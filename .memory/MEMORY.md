# Repository Memory

## Stable Context
- 龍蝦群目前能確認的長期穩定脈絡很少；這個 repo 的記憶仍在非常早期的建立階段。
- 已知主人會先把事情放進 issue 系統作為協作入口，即使需求尚未整理完整，也會先留下一個可追蹤節點，避免工作線索散失。
- 目前可見的協作模式偏向 **主人 + Copilot** 共同掛名處理事項，表示這個 repo 預設接受人機協作；但實際分工方式、責任邊界與交付流程，尚未形成穩定規則。
- 從現有資料看不出明確的產品領域、技術棧、開發慣例或發布節奏；這些都仍屬未知，不能過早寫成穩定事實。

## Recent Themes
- **先佔位、後澄清需求**  
  最近最明顯的工作節奏，是先建立 issue 作為入口，再等待後續補齊背景、目標與成功條件。這比較像是把模糊想法先收進系統，而不是立即進入實作。
- **協作關係已存在，但任務定義不足**  
  目前唯一可見的工作支線已指派給主人與 Copilot，顯示協作意圖明確；但誰負責定義問題、誰負責執行、誰負責驗收，還沒有被寫清楚。
- **工作內容仍高度不確定**  
  目前無法判斷最新支線屬於產品功能、工程維護、流程優化，還是其他生活／工作管理事項。龍蝦群只能先記住：有一件事被正式掛進 repo，但尚未成形。

## Constraints
- **資訊來源不足**  
  目前記憶主要來自極少量 issue 資訊，且內容接近空白；因此不能把單日觀察誤判為長期規則。
- **需求未明前不宜展開實作**  
  在缺少背景、範圍、目標與驗收標準的情況下，任何實作都容易偏題。這是目前最重要的工程限制。
- **repo 脈絡尚未成熟**  
  還看不出固定標籤策略、issue 模板、優先級規則、分支流程或文件慣例；若後續要擴充協作，這些都可能需要補建。
- **不確定性需明確保留**  
  由於現有資料不足，龍蝦群應持續把未知標成未知，而不是替主人腦補任務性質或決策背景。

## Open Loops
- **Issue #1 需要補上完整內容**  
  目前只知道它存在、已指派、可追蹤，但缺少真正的標題意義、任務描述與預期成果。
- **需要補需求背景與來源**  
  尚不清楚這件事是來自臨時想法、外部需求、對話紀錄，還是其他工作脈絡。若不補來源，後續很難判斷優先級與範圍。
- **需要定義成功條件與驗收方式**  
  目前沒有完成標準，無法判斷何時算做完，也無法有效分工給主人或 Copilot。
- **需要釐清協作分工**  
  雖然主人與 Copilot 都被指派，但尚未說明誰負責需求整理、誰負責執行、誰負責最後確認。
- **需要更多 repo 級脈絡**  
  目前仍缺少足夠資料來建立穩定的長期記憶，例如技術方向、常用工作流、文件習慣與常見限制。
