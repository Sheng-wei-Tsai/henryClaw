# Repository Memory

## Stable Context
- 龍蝦群目前能確認的長期脈絡非常少；這個 repo 在最近可見記憶中，**沒有穩定浮現的產品主線、技術架構演進、維運節奏或生活事件串接**。
- 唯一反覆被確認的穩定事實是：**Issue #1 是一張缺乏需求、背景、討論與產出的占位／測試性 issue，且已於 2026-05-15 關閉。**
- 對這個 repo 的記憶整理方式偏向保守：**沒有證據就不補故事，沒有上下文就不把空白票據當成有效待辦。**
- 目前看來，龍蝦群替主人維護這個倉庫記憶時，重點不是追逐表面上的任務數量，而是**維持記錄可信度與任務邊界清楚**。

## Recent Themes
- 最近幾天反覆出現的主題，不是功能開發，而是**記憶衛生／issue hygiene**：
  - 清點哪些 issue 其實沒有形成真正工作。
  - 避免把空白或測試性節點誤記成待辦。
  - 在資料不足時明確承認「目前不知道」。
- `#1` 被多次重新確認為：
  - 無實質需求內容
  - 無留言或討論脈絡
  - 無 Telegram 或其他外部上下文可還原
  - 無實作、驗收或可延續成果
- 因此，近期這個 repo 的主要動作比較像是**清場與除噪**，而不是推進新功能、修 bug 或做規格決策。
- 也要誠實記下不確定性：**目前資料量過低**，所以「沒有看到進展」不等於「主人完全沒有在做事」；只能說，在這批 memory 來源裡，尚未留下可可靠整理的工程脈絡。

## Constraints
- **不可腦補需求。** 若 issue 缺少目標、背景、成功條件與討論來源，龍蝦群不應自行延伸成任務。
- **GitHub issue / comment 才是原始資料來源。** 手動筆記與 memory 彙整應該提煉，不應複製原始 issue 內容。
- **空白 issue 不視為有效 backlog。** 缺乏需求定義的項目，不應被當成可執行工作。
- **記憶整理以保守歸檔為原則。** 在證據不足時，優先標記為資訊不足、停止追蹤，而不是製造虛假的進度敘事。
- **目前 repo 上下文稀薄。** 因缺少更多 issue、PR、commit 或外部對話脈絡，很多高階判斷都只能維持低信心。

## Open Loops
- **#1 若未來要重啟，需要主人先補齊：**
  - 明確目標
  - 背景脈絡
  - 想解決的問題
  - 預期產出或完成條件
- **若 #1 曾有外部討論但尚未匯入**（例如 Telegram 或其他渠道），仍待補齊，否則無法重新判定它是否值得復活。
- **整體 repo 的主線仍不明。** 目前無法確認：
  - 是否存在未被記錄的專案主題
  - 是否有 repo 外工作正在承接這裡的脈絡
  - 是否只是目前 issue 記錄密度過低
- 若主人希望未來龍蝦群能保留更有用的長期記憶，後續仍待補強：
  - issue 的需求背景
  - 決策理由
  - 預期驗收方式
  - 與外部對話或實作成果的連結
