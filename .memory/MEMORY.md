# Repository Memory

## Stable Context
- 🦞 這個 repo 目前可確認的長期脈絡非常少；近 30 天可見的 issue 記憶幾乎只剩 **#1**，而它已被反覆確認為**空白占位／測試性質的歷史紀錄**，不是可延續的工程任務。
- 🦞 **#1 已於 2026-05-15 關閉**，目前不應再被視為活躍工作流的一部分，也不應拿來推導功能需求、產品方向或技術決策。
- 🦞 龍蝦群在這個 repo 的共同整理原則很明確：**沒有需求描述、背景脈絡、討論紀錄或驗收條件，就不把 issue 當成有效待辦**。
- 🦞 這個 repo 目前更像是在做**記憶衛生與任務邊界管理**，而不是有穩定可追的多線開發節奏。
- 🦞 依照 shared manual notes 的精神，長期記憶應保留：
  - 穩定規則
  - 長期決策
  - 常見限制
  - agent 共同遵守的 repo 習慣  
  而不是複製原始 issue 內容。

## Recent Themes
- 🦞 最近幾天反覆出現的主題不是新功能，而是**清理 backlog 雜訊**：龍蝦們持續確認哪些卡片其實沒有內容，避免主人把空殼誤認成待辦。
- 🦞 **資料品質不足** 是近期最穩定的訊號。現有 issue 記錄缺少：
  - 任務目標
  - 背景脈絡
  - 討論過程
  - 驗收條件
  - 實作或交付紀錄
- 🦞 **保守整理、不腦補需求** 是近期反覆被採用的工作方式。對於像 #1 這種資訊稀薄的卡片，龍蝦們選擇歸檔，而不是替主人虛構下一步。
- 🦞 從近期 snapshots 看來，這段時間較像是**盤點與收束追蹤系統**，不是明確的產品推進期。
- 🦞 目前沒有足夠證據顯示 repo 內存在其他穩定主題、技術方向或協作節奏；若主人其實有在做別的事，至少**沒有被這批 issue 記憶可靠地記下來**。

## Constraints
- 🦞 **資訊來源受限**：目前可用記憶主要來自 issue 摘要；若沒有 issue 內容、留言或外部對話摘要，龍蝦群就無法可靠還原工作脈絡。
- 🦞 **不可過度推論**：不能只憑 issue 標題或存在與否，就推定有需求、規格或實作計畫。
- 🦞 **空白 issue 不可作為開發依據**：像 #1 這類缺乏內容的卡片，只能視為占位紀錄，不能當成 backlog 項目。
- 🦞 **可追溯性偏低**：目前看不到 Telegram 或其他外部討論摘要是否存在；若有，尚未進入這份記憶鏈。
- 🦞 **現況存在不確定性**：repo 近況可能只是「沒有被記錄」，不一定等於「完全沒有工作發生」。因此任何「目前沒有進展」的判斷，都應理解為**在現有資料範圍內沒有可證實進展**。

## Open Loops
- 🦞 **目前沒有明確仍在進行中的工程事項**；至少在現有記憶中，看不到可直接承接的 active issue。
- 🦞 若主人未來想重啟 **#1** 背後的方向，仍需先補齊：
  - 任務目標
  - 背景／問題定義
  - 範圍
  - 預期成果
  - 驗收條件
  - 相關討論紀錄
- 🦞 對於「應該重新開新 issue」還是「補齊後沿用 #1」這件事，近期記憶中有兩種說法；較穩妥的結論是：**不要直接沿用目前這張空白卡片開始做事，至少要先補完整上下文，必要時另開新 issue 以避免混淆。**
- 🦞 另一個未解開口是：**是否有 repo 外的工作脈絡尚未被納入記憶系統**。如果主人有透過 Telegram、口頭或其他地方討論過相關事項，龍蝦群目前還沒拿到可整理的摘要。
