# Repository Memory

## Stable Context
- 龍蝦群目前能穩定確認的 repo 現況很單純：這個 repository 的 **GitHub issue 系統已被當作長期記憶與工作追蹤入口**，但可見內容仍非常稀薄。
- 近一段時間內，唯一反覆出現且可辨識的節點是 **Issue #1**：
  - 已建立且保持開啟。
  - 已指派給 **主人本人與 Copilot**。
  - 目前仍缺乏足夠描述，無法判定其屬於產品開發、技術修補、流程整理、生活管理，或其他類型任務。
- 從多日快照反覆一致的訊號來看，主人似乎有一種工作習慣：**先建立 issue 容器／占位，再視情況補內容**。  
  但這目前只能算「傾向」，不是完全穩定規則，因為樣本幾乎只有 #1。
- 龍蝦群在整理時採取保守原則：**當 issue 缺乏正文、背景與驗收標準時，不替主人腦補任務內容**，避免把猜測寫成記憶。

## Recent Themes
- 最近最明顯的主題不是功能開發，而是 **「記錄結構已存在，但內容尚未沉澱」**。
- 多日快照都指出同一個模式：
  - 有追蹤入口（Issue #1）。
  - 有協作者配置（主人 + Copilot）。
  - 但沒有任務目標、背景脈絡、成功條件、留言討論或可執行拆解。
- 這代表目前 repo 的主要問題不是技術阻塞，而是 **上下文缺失**：
  - 外部討論（尤其多次被提到的 Telegram 脈絡）尚未回填。
  - 即使有 issue，也還不足以讓其他 agent 或未來的主人自己順利接手。
- 近期反覆浮現的另一個主題是：**知識匯入鏈條不完整**。  
  若主人確實有在其他地方推進事情，repo 目前看不見，因此龍蝦群也無法替主人保存那段記憶。
- 整體來說，最近的 repo 狀態比較像是：
  - 已開始搭記憶骨架；
  - 尚未形成可協作、可追蹤、可驗收的工程敘事。

## Constraints
- **資訊極度不足**：目前幾乎所有可見脈絡都圍繞在空白的 Issue #1，沒有足夠資料支撐更大的專案判讀。
- **不可過度推論**：由於 #1 缺少標題語意、正文、留言與驗收標準，不能可靠推定其任務類型、優先順序或技術方向。
- **外部脈絡未同步**：多份快照都提到可能存在 Telegram 或其他外部討論，但尚未匯入 repo；這使記憶鏈容易斷裂。
- **可接手性偏低**：目前 issue 記錄不足以支援代理人協作、後續拆解或進度追蹤。
- **樣本過少，穩定規則不宜寫死**：目前觀察到的習慣大多來自單一 issue，仍有不確定性。

## Open Loops
- **Issue #1 到底要解決什麼問題？**  
  目前仍無法判定其主題、角色與預期產出。
- **需要補上 #1 的基本任務定義**：
  - 明確標題
  - 背景脈絡
  - 任務目標
  - 完成條件
  - 優先順序或範圍
- **需要回填外部上下文**：若 #1 來自 Telegram、口頭討論或其他筆記，應整理關鍵內容進 issue。
- **需要釐清協作分工**：雖然 #1 已指派給主人與 Copilot，但目前不知道誰主導、誰執行、誰審核。
- **需要確認真實進度**：不確定主人是否已在 repo 外推進相關工作，只是尚未回寫；若是，應補記錄，若否，則要評估 #1 是否只是空殼占位。
- **需要判斷 #1 是否應重寫或重新命名**：若長期沒有內容，可能需要整理成真正可執行的 issue，而不是持續保留模糊入口。
