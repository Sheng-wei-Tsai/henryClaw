# Repository Memory

## Stable Context
- 龍蝦群目前能穩定確認的 repo 脈絡很少；這個記憶洞穴在最近可見資料中，**主要用途偏向協作記憶與工作掛點**，而不是已有明確產品功能線正在持續推進。
- 已知存在一個長期被反覆提及的入口：**Issue #1**。它目前較像是：
  - 記憶容器
  - 後續協作入口
  - 等待主人補充內容的占位節點
- 從多日快照可蒸餾出的穩定工作習慣是：**先保留可接手的記憶結構，再等待需求與上下文補齊**；龍蝦群傾向整理濃縮後的可交接記憶，而不是保存逐字聊天原文。
- 目前**沒有足夠證據**可確認此 repo 的穩定產品方向、技術棧主線、開發流程細節或固定維運節奏；若要寫成規則，會超出現有資料支持範圍。

## Recent Themes
- 最近數日的共同主題非常一致：**資訊不足比技術問題更核心**。
- 在 2026-04-22 到 2026-04-24 的快照中，龍蝦群反覆確認：
  - `#1` 已存在且為 open
  - 已指派給主人與 Copilot
  - 但缺少足夠內容，無法形成可執行任務
- 2026-04-25 到 2026-04-28 的整理視窗則幾乎沒有可用 issue；代表近期 repo 活動非常安靜，**沒有新的跨議題主線浮現**。
- 可重複觀察到的近期模式：
  - **記憶基礎設施先行**：先有 issue 掛點，再補任務內容。
  - **避免過度推論**：在需求未明前，不拆任務、不定優先級、不假設技術方案。
  - **需要可交接脈絡**：主人似乎重視後續能接手、能追蹤、能延續，而不是零散對話本身。
- 不確定性：這些主題目前主要來自單一 issue 與多日「無新資料」的重複觀察，**可視為近期趨勢，但還不足以證明是長期不變的 repo 策略**。

## Constraints
- **資料來源限制明顯**：目前 issue 與快照內容極少，龍蝦群無法可靠重建主人實際做過的工作、決策或生活／工作安排。
- **不可把占位 issue 誤寫成已定義任務**：`#1` 雖然存在，但現階段不能視為已有明確需求、驗收條件或實作方向。
- **缺少上下文附件**：目前沒有足夠的留言、Telegram 對話、背景說明或決策紀錄可供交叉驗證。
- **不適合排程與估工**：在目標、範圍、成功條件未明前，任何優先順序、工時或技術拆解都不可靠。
- **近期觀測面偏窄**：最近多日甚至沒有可用 issue，因此任何「repo 正在做什麼」的結論都應保守處理。

## Open Loops
- **Issue #1 仍待補完核心資訊**：
  - 目標是什麼
  - 背景脈絡是什麼
  - 預期輸出／成果是什麼
  - 完成條件如何判定
  - 這隻龍蝦／Copilot／主人各自扮演什麼角色
- **需要補入原始脈絡**：若有 Telegram 對話、issue 留言、需求描述或其他決策紀錄，應匯入或摘要，否則記憶無法穩定延續。
- **需要重新確認 repo 是否有其他工作流**：目前快照只反映 `#1` 與多日無新 issue；若主人其實有其他進展，現有記憶尚未捕捉到。
- **等待下一輪有效 issue 更新**：在沒有新資料前，龍蝦群只能保留這個最小記憶骨架，暫不擴寫成更強的長期結論。
