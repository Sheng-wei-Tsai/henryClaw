# Repository Memory

## Stable Context
- 龍蝦群目前能穩定確認的 repo 脈絡非常少；這個倉庫在近一段時間內，**幾乎只有一條可見工作入口：Issue #1**。
- 主人似乎有一種工作習慣：**先在 GitHub issue 開一個入口或占位，再視情況補內容**。但在這個 repo 裡，這個習慣目前只看到單一案例，是否為長期穩定流程，仍有些不確定。
- `#1` 已被指派給**主人本人與 Copilot**，可推知這個 repo 預期支援**人機協作**，至少 issue 層級會把 Copilot 視為可參與的協作者。
- 目前沒有足夠證據判定此 repo 的主題屬於產品開發、技術債整理、流程管理，或生活任務管理；**任務領域仍不明**，不能自行腦補。
- 龍蝦群在這個 repo 的長期記憶策略應偏保守：**當 issue 缺乏標題、內文、留言與外部脈絡時，不把它當成已定義任務，只把它視為待補資訊的記憶鉤子**。

## Recent Themes
- 最近反覆出現的主題非常一致：**記錄容器已建立，但內容沒有進場**。
- `#1` 持續處於「開著、已指派、但幾乎空白」的狀態；這表示目前最大的阻塞不是實作能力，而是**需求定義與上下文缺失**。
- 龍蝦群這幾天的主要工作，不是在推進功能，而是在替主人守住一條重要原則：**資訊不足時不推測、不補完、不製造假記憶**。
- 另一個反覆浮現的主題是：**外部討論沒有回填進 repo**。多份快照都提到可能缺少 Telegram 或其他來源的背景，導致 issue 無法成為可接手的工程記憶節點。
- 整體來看，這個 repo 近期更像是在做**記憶治理與 issue hygiene**，而不是進入穩定的開發節奏。
- 目前也反覆出現一個風險訊號：若主人持續採用「先開入口、後補內容」但沒有回填，repo 會累積**開著但不可執行的空殼 issue**。

## Constraints
- **不可臆測需求內容。** `#1` 沒有足夠資訊，不能假設它是功能開發、文件整理、研究任務或生活提醒。
- **GitHub issue/comment 才是主要可追溯來源。** 若背景只存在於 Telegram、口頭討論或其他外部媒介，而未回填到 repo，龍蝦群就無法可靠保存與延續。
- **目前缺少完成條件、優先順序、範圍與責任邊界。** 在這些資訊缺席時，任何拆解、排序或執行建議都容易失真。
- **可觀測資料量極低。** 近 30 天快照幾乎只圍繞 `#1`，因此不能從單一樣本過度推導主人的長期工作模式。
- **沒有標籤、沒有留言、沒有明確標題語意。** 這使得 issue 幾乎無法被自動摘要、分類或串聯成更大的專案脈絡。
- 若主人其實已在其他地方推進工作，但**沒有寫回 issue**，那麼對龍蝦群而言，這些進展目前等同不可見。

## Open Loops
- `#1` 需要補上**明確標題與 issue 內文**：到底要解什麼問題、為誰解、為何現在要做。
- 需要補充**背景脈絡來源**：若此事來自 Telegram、口頭討論、臨時想法或其他筆記，應整理回 issue。
- 需要定義**預期產出與完成條件**：什麼算完成、是否需要拆子任務、是否有驗收標準。
- 需要釐清**優先順序與責任分工**：主人主導、Copilot 協作，還是只是先占位等待未來處理。
- 需要確認 `#1` 是否真的是有效待辦；若只是暫存想法，可能要**重寫、重新命名，甚至關閉**，避免長期佔著 open 狀態卻無法行動。
- 目前仍不確定這個 repo 的核心用途與工作邊界；在更多內容出現前，龍蝦群只能把它記成：**有協作入口，但尚未形成可執行任務的倉庫**。
