# Repository Memory

## Stable Context
- 龍蝦群目前能確認的長期脈絡很少；這個 repo 在最近可見的 issue 視窗中幾乎沒有持續性的工程討論、模組脈絡或產品方向可供掛鉤。
- 目前唯一較穩定可提煉的協作原則是：**issue 必須帶有足夠背景、目標與完成條件，才值得持續追蹤**。沒有內容的殼不應長期佔住注意力。
- 從最近整理結果看來，主人傾向讓追蹤面板保持乾淨：若某個 issue 只是占位、測試或缺乏實質需求，會直接收束，而不是勉強延續。
- `.memory/shared/manual.md` 的角色是**人工維護的長期記憶**：適合放穩定規則、長期決策、常見限制與共同習慣；它不是原始資料來源，也不應複製完整 issue 內容。

## Recent Themes
- **清理追蹤噪音**：2026-05-15 唯一明確事件是 issue #1 被關閉，脈絡顯示它較像占位／測試用紀錄，而非真正承載需求的任務。
- **要求任務描述更完整**：最近唯一可見的整理結論，是未來若要讓龍蝦群接手記憶與協作，開 issue 時最好補上背景、目標、驗收條件，避免再次出現只有殼、沒有內容的條目。
- **整體活動量偏低**：2026-05-09 到 2026-05-14 幾乎都沒有可用 issue；目前看不出穩定的開發主線、除錯節奏或文件整理波段。
- **不確定性高**：缺乏 issue 不代表 repo 完全沒有工作，只代表最近沒有足夠進入 issue 軌道的可見脈絡。

## Constraints
- **資料來源限制**：目前記憶主要依賴 GitHub issue / comment；若工作沒有進入 issue 脈絡，龍蝦群就難以替主人穩定掛回長期記憶。
- **資訊稀少**：最近 30 天內可用資料極少，因此不應過度推論 repo 的產品方向、技術棧或工作習慣。
- **記憶整理原則**：這份 MEMORY 應保留蒸餾後的長期脈絡，而不是逐日摘要或 issue 索引。
- **追蹤成本控制**：對缺乏背景、目標與驗收條件的 issue，不宜投入持續追蹤成本；若要重啟相關想法，較適合另開新 issue 並補齊上下文。

## Open Loops
- 是否需要建立或強化 **issue 模板**，明確要求填寫背景、目標、完成標準，避免再出現空殼 issue。
- 若已關閉的 **#1** 背後其實有真實想法，目前仍是未定義狀態；需要主人日後以新 issue 重新描述需求，而不是沿用舊殼。
- 目前沒有足夠資訊判斷 repo 是否存在其他進行中的工作流、阻塞或長期目標；這部分仍待未來 issue 活動補齊。
- 由於近期資料過少，**Stable Context 仍偏薄**；後續若出現連續數日可串聯的議題，再考慮升格為穩定規則或長期主題。
