# Repository Memory

## Stable Context
- 龍蝦群目前能穩定確認的 repo 現況很少：`Sheng-wei-Tsai/henryClaw` 在最近可見視窗內，**幾乎沒有可讀的持續性工作脈絡**。
- 唯一反覆出現且可辨識的節點是 **Issue #1**：
  - 已建立
  - 仍為 open
  - 曾被指派給主人與 Copilot
- 從多日快照一致可見，這個 repo 目前比較像是**先建立追蹤容器，再等待內容補齊**的狀態；也就是說，追蹤入口存在，但需求、背景、驗收與進度尚未穩定寫入。
- 龍蝦群已形成一條可重用的記憶原則：**在 issue 缺少標題語意、內文、留言與外部脈絡時，不替主人腦補任務內容**，避免把猜測誤記成長期事實。
- 目前沒有足夠證據判定這個 repo 的主要用途是產品開發、生活管理、研究實驗，或其他工作流；這點仍屬**不確定**。

## Recent Themes
- 近幾天最明顯的主題不是功能推進，而是**資訊缺口本身**：
  - 有 issue 容器
  - 有協作指派
  - 但沒有足夠上下文讓人或 agent 接手
- **#1 長期呈現「空殼 issue」狀態**：它像是一個被保留下來的掛點，代表主人可能有先記錄入口的習慣，但尚未把真正需求落盤。
- **協作意圖存在，但協作契約未落地**：主人與 Copilot 被掛在 #1 上，顯示原本可能期待人機協作；然而目前沒有可執行分工、沒有完成標準，也沒有進度回填。
- **repo 外脈絡可能存在，但尚未回填**：多日摘要都提到 Telegram 或其他外部對話可能是需求來源，表示真正的上下文可能散落在 repo 外。
- 整體來看，近期不是「某項工作卡住」，而是**連要做什麼都還沒被正式定義清楚**。

## Constraints
- **資料極度不足**：目前幾乎只能依賴 #1 的 metadata，缺少標題語意、描述、留言、標籤與可驗證進展。
- **不可過度推論**：在沒有明確內容前，不能把 #1 視為已定義任務，也不能推導其優先順序、技術方案或產品意圖。
- **記憶來源受限於 repo 內可見資訊**：若主人實際已有進展，但未寫回 issue，龍蝦群就無法可靠保存這段記憶。
- **外部對話未同步是主要風險**：若需求背景主要存在 Telegram 或其他渠道，repo 內記憶會持續失真或斷裂。
- **目前沒有足夠證據建立穩定 repo 規範**：除了「不要臆測空白 issue」之外，尚看不出更成熟的工程慣例或流程規則。

## Open Loops
- **#1 需要補齊核心定義**：
  - 這個 issue 到底要解決什麼問題
  - 屬於哪種情境（生活／工作／產品／研究／測試）
  - 為什麼現在要做
- **需要補上最小可執行資訊**：
  - 任務目標
  - 背景脈絡
  - 預期成果
  - 完成條件／驗收標準
  - 下一步行動
- **需要確認協作角色**：Copilot／agent 在 #1 中是執行者、整理者、提醒者，還是僅作為備援協作者。
- **需要回填 repo 外上下文**：若曾在 Telegram 或其他地方討論過 #1，應整理摘要寫回 issue，否則這條記憶支線會一直停留在空殼狀態。
- **需要確認是否已有隱含進展未記錄**：目前無法判斷主人是否私下已推進，只是尚未同步到 repo。
- **不確定性仍高**：截至目前，龍蝦群只能確定「有一個待補內容的 open 容器」，不能確定其真正任務性質。
