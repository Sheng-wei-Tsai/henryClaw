# Repository Memory

## Stable Context
- 龍蝦群目前能穩定確認的 repo 脈絡很少；這個倉庫仍處於**記憶基礎設施先建立、實際任務內容後補**的早期階段。
- 已知存在一條核心入口：**Issue #1**。它長期維持為一個**協作掛點／記憶容器**，而不是已明確定義的功能任務。
- 主人似乎接受一種工作方式：**先開 issue 保留入口，再逐步補上需求、背景與對話脈絡**。
- 另一個可重用的穩定偏好是：主人傾向把協作內容整理成**可交接、可濃縮的工作記憶**，而不是保存逐字聊天紀錄。這與 shared manual notes 的方向一致。
- 目前沒有足夠證據支持任何產品方向、技術棧選型、模組邊界、生活流程規則或排程慣例；若要寫成穩定規則，會超出現有資料。

## Recent Themes
- 最近反覆出現的主題幾乎都圍繞在 **#1 的資訊不足**：
  - issue 已存在、已開啟；
  - 主人與 Copilot 被視為參與者；
  - 但缺少標題語意、內文、留言、外部對話與成功條件。
- 龍蝦群這幾天的主要工作不是推進實作，而是**避免誤判**：先把空白支線標記為待補資料，而不是替主人腦補需求。
- 一個持續浮現的訊號是：**工程可讀性／可接手性** 被重視。也就是說，這個 repo 的近期重點比較像是在建立「未來能接手的記憶格式」，而不是交付具體功能。
- 目前觀測到的阻塞不是技術難題，而是**任務定義缺失**：不知道要做什麼、為何而做、做到哪裡算完成。
- 不確定性很高：若主人在這段期間其實有其他工作或生活進展，現有 issue 快照沒有留下足夠痕跡，龍蝦群無法可靠重建。

## Constraints
- **不可臆測需求**：在缺少 issue 內容、留言或外部對話前，不應自行定義 #1 的功能、優先級或解法。
- **目前只能依賴稀薄的 issue metadata**：可確認的多半只有 open 狀態、參與者與「內容不足」這件事本身。
- **記憶整理應以蒸餾為主，不是逐字存檔**：這是 shared manual notes 與近期快照共同指向的協作原則。
- **缺乏可追溯來源** 是當前主要限制：沒有 Telegram 對話、留言紀錄或需求背景時，後續代理難以接手。
- **不適合估工、排程或拆解任務**：在完成條件與目標未明前，任何細部規劃都容易偏離主人原意。
- **資料覆蓋面不足**：目前近 30 天可見內容幾乎只剩 #1，因此任何「repo 已形成穩定工作流」的判斷都應保守看待。

## Open Loops
- **#1 的核心任務仍未定義**：這條支線到底要處理什麼類型的工作、問題或生活脈絡，尚不清楚。
- **需要補齊最小可執行上下文**：
  - 任務目標；
  - 背景脈絡；
  - 預期輸出；
  - 完成條件；
  - 優先順序。
- **需要補入來源材料**：尤其是相關留言、Telegram 對話或其他需求紀錄，讓記憶鏈可追溯。
- **需要重新判定 #1 的角色**：它究竟只是占位容器，還是將成為某條長期協作主線，目前仍未明。
- **需要觀察是否會出現更多 issue 或實際內容**：若後續仍只有空白入口，這個 repo 會持續停留在「有容器、無任務」的狀態。
- **不確定性聲明**：目前沒有足夠資料判定主人是否已有其他隱含決策、技術方向或生活安排；龍蝦群暫時只能誠實記住「資訊空缺本身」。
