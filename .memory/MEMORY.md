# Repository Memory

## Stable Context
- 龍蝦群目前能確認的長期穩定脈絡很少；這個 repo 在近 30 天內可見的工作記錄幾乎只圍繞 **單一 issue 容器 `#1`**。
- 主人似乎有一種工作習慣：**先把事情放進 issue 系統佔位，再逐步補內容**。至少在 `#1` 上，已先建立追蹤入口並完成指派，但需求定義尚未成形。
- `#1` 已指派給 **主人與 Copilot**，可視為目前唯一明確可見的人機協作節點；不過這只代表協作框架存在，**不代表分工、目標或實作方向已經確立**。
- 從 shared manual notes 可知，這個記憶系統的長期原則是：
  - GitHub issue / comment 才是原始資料來源。
  - 長期記憶應保留穩定規則、長期決策、常見限制與 repo 習慣。
  - 不應把 issue 原文整段複製進記憶文件。
- 目前沒有足夠證據判定此 repo 的主要用途是產品開發、流程工具、Telegram 整理，或個人生活管理；**用途仍不確定**。

## Recent Themes
- 最近反覆出現的核心主題只有一個：**工作入口已建立，但語意與背景嚴重不足**。
- `#1` 在多日快照中都呈現相同模式：
  - 標題仍接近預設占位狀態。
  - 缺少正文、留言與需求背景。
  - 看不出實質推進、設計討論或交付成果。
- 另一個反覆浮現的主題是：**知識匯入鏈條尚未完成**。
  - 多份快照都提到看不到 Telegram 或其他外部對話脈絡。
  - 這表示主人可能有外部討論或想法來源，但尚未穩定轉譯進 repo。
- 近期的實際進展，不像是功能開發，而比較像是：
  - 先保留一個不讓事情蒸發的追蹤入口。
  - 再等待主人補齊背景、目標與完成條件。
- 因為近幾天資訊高度重複且單薄，龍蝦群暫時**不把這些訊號升格為更廣泛的工程流程規則**；目前只能說這是近期明顯模式，不一定是長期制度。

## Constraints
- **資訊不足是目前最大的限制。**
  - 只有 `#1` 可見，且內容近乎空白。
  - 沒有足夠上下文可做需求拆解、優先級排序或技術判斷。
- **不能替主人腦補任務內容。**
  - 在缺少正文、留言、來源對話與驗收標準時，龍蝦群應避免推測 `#1` 的實際範圍，避免污染長期記憶。
- **外部脈絡未匯入 repo 時，記憶系統無法保存。**
  - 若工作來自 Telegram、臨時想法或口頭討論，但沒有整理進 issue/comment，後續代理與未來的主人自己都難以接手。
- **目前看不到穩定標籤與分類機制。**
  - 近幾日快照皆顯示 `Top Labels: none`，代表 repo 尚未形成可見的 issue 分類語彙，或至少目前資料不足以觀察到。
- **協作存在但分工不明。**
  - 雖然 `#1` 指派給主人與 Copilot，但誰負責定義需求、誰負責實作、誰負責驗收，仍未寫明。
