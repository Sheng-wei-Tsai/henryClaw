# Repository Memory

## Stable Context
- 龍蝦群目前能確認的穩定脈絡很少：這個 repo 在最近可見記憶中，**沒有持續活躍的 issue 流**，也沒有可反覆驗證的功能開發主題、修 bug 節奏或文件維護慣例。
- 目前唯一較明確的長期可用規則，是偏向**維持待辦池乾淨、避免空白任務污染注意力**：
  - 沒有描述、沒有討論、沒有範圍的 issue，不應被當成有效工作入口。
  - 若某條任務只是占位、且未形成實質需求，應傾向關閉，而不是讓它長期停留在 backlog 中。
  - 若未來要重啟已被清掉的方向，應以**新 issue 重新定義目標、範圍與預期產出**，不要沿用缺乏上下文的舊殼。
- 手動筆記本身也透露一條 repo 級習慣：**GitHub issue / comment 才是原始事實來源**；記憶文件應該是整理後的長期脈絡，不是逐字備份。

## Recent Themes
- 最近唯一可辨識的主題是**任務治理與資訊衛生**，而不是產品功能推進。
- 2026-06-13 的快照顯示，`#1` 被判定為**無實質內容的占位 issue**，並已關閉。這代表近期曾做過一次 backlog 去噪：把沒有需求內容的支線正式收掉。
- 2026-06-14 到 2026-06-19 的多日快照都顯示：
  - 沒有新的可用 issue 資料。
  - 沒有新的跨 issue 主題。
  - 沒有新的跨 issue 決策。
- 因此，近期狀態比較像是**低活動或無活動期**。龍蝦們目前看不到足夠證據，能把任何新方向寫成穩定事實。

## Constraints
- **資訊來源不足**：最近多日快照都沒有可用 issue，代表目前記憶基礎非常薄，很多判斷只能停留在「暫時看不到」，不能推論成「確定不存在」。
- **不可把單日事件誤寫成長期規則**：除了 `#1` 的清理案例外，沒有更多樣本支持更強的流程結論。
- **原始資料依賴 GitHub issue/comment**：若 issue 本身沒有描述，龍蝦群就無法可靠還原主人的原始意圖。
- **目前沒有標籤、主題串或多 issue 關聯可供萃取**：這限制了跨任務層級的 repo context 建立。
- **存在不確定性**：目前無法判斷 repo 是真的暫停中、只是近期未用 issue 記錄工作，或工作脈絡存在於其他未提供來源。

## Open Loops
- 主人是否還記得 `#1` 原本想承載的方向？如果記得，現有資料已不足以延續，應重新開立新 issue 並補齊背景。
- 這個 repo 目前是否有**未透過 issue 記錄**的實際工作流、需求來源或維護節奏？現有記憶無法確認。
- 等待下一輪有內容的 issue 出現後，龍蝦群才能補齊：
  - repo 的實際功能目標
  - 常見工作類型
  - 穩定協作習慣
  - 真正的未完成事項與依賴關係
- 在新資料出現前，最安全的結論是：**目前沒有可確認的活躍工程 open loop，只有記憶來源過少這件事本身仍未解。**
