# Repository Memory

## Stable Context
- 龍蝦群目前能穩定確認的 repo 現況很少：**這個 repository 已開始用 GitHub issue 承接記憶／任務入口，但可見內容仍極度稀薄**。
- 近一段時間內，唯一反覆出現且可辨識的工作節點是 **Issue #1**；它長期維持 open，並被視為一個**保留中的入口／容器**，而不是已明確定義的可執行任務。
- `#1` 已指派給 **主人 (`Sheng-wei-Tsai`) 與 Copilot**，可推知 repo 預期存在某種**人機協作**模式；但目前協作契約尚未寫清楚，不能把這件事視為已落地的流程。
- 龍蝦群在整理時採取明確保守原則：**若 issue 缺少標題語意、內文、留言與背景脈絡，就不臆測需求、不虛構進度、不替主人腦補任務內容**。
- 手動筆記的長期規則也很清楚：**GitHub issue / comment 才是原始資料來源**；長期記憶應該提煉穩定規則與限制，而不是複製原始 issue 內容。

## Recent Themes
- 最近最反覆出現的主題只有一個核心：**有記憶容器，但沒有足夠上下文讓代理真正開工**。
- `#1` 一直呈現「**先建立入口、內容後補**」的狀態：  
  - issue 存在  
  - 指派存在  
  - 但任務目標、背景、完成條件、優先順序都缺席
- 龍蝦群近期的工作重心不是功能推進，而是**資訊治理與記憶止血**：先確認哪些資訊不存在，避免把空白殼誤記成已定義任務。
- 多日摘要都反覆指出：**repo 外部可能有需求來源**，例如 Telegram 或其他對話，但這些脈絡尚未回填到 issue，因此記憶鏈條是斷的。
- 目前無法可靠判定 `#1` 屬於：
  - 生活待辦
  - 工作任務
  - 產品／工程支線
  - 單純測試或占位 issue  
  這個不確定性本身，就是近期最重要的事實。

## Constraints
- **資訊不足是目前最大的硬限制。** 在沒有標題、內文、留言、外部對話摘要前，任何需求拆解、優先排序或技術判斷都不安全。
- **不可把 open 視為「正在執行中」。** 目前 `#1` 的 open 更像是「尚未定義完成」，不是可證明的執行進度。
- **不可把指派視為已完成分工。** 雖然主人與 Copilot 已被指派，但 agent 的角色究竟是執行者、整理者、提醒者或備援協作者，仍未定義。
- **repo 內缺乏可交叉驗證的 issue 網絡。** 目前幾乎只有 `#1`，因此無法建立更高層的產品方向、模組關聯或生活／工作節奏圖。
- **外部脈絡未回填會直接削弱工程可讀性。** 若需求與決策散落在 Telegram 或其他地方，龍蝦群只能記住「可能有背景」，不能記住背景內容本身。
- **目前沒有足夠證據形成穩定 repo 慣例。** 除了「保守、不臆測」之外，尚看不出更成熟的工作流規則。

## Open Loops
- **`#1` 的真實定位仍未明。** 需要確認它到底代表哪一條支線：生活、工作、產品、工程，或只是測試用容器。
- **需要補齊 `#1` 的最小可執行資訊：**
  - 任務目標
  - 背景脈絡
  - 預期成果
  - 完成條件
  - 優先順序
- **需要確認 repo 外是否已有原始脈絡**，例如 Telegram 對話、留言、觸發事件或需求來源，並回填到 issue。
- **需要定義 Copilot / agent 在 `#1` 的角色邊界**：主動執行、協助整理、提醒追蹤，還是僅在主人補完需求後才介入。
- **需要確認是否存在隱含但未寫出的限制**，例如截止時間、依賴項、輸入輸出格式或驗收方式。
- **需要觀察這種「先開 issue 再補內容」是否會成為穩定習慣。** 目前它只在近期反覆出現，但還不足以升格為長期規則。
