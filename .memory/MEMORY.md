# Repository Memory

## Stable Context
- 龍蝦群目前能穩定確認的 repo 現況很少：**近一段時間可見的工作脈絡幾乎只圍繞在 Issue `#1`**。
- `#1` 是一張**已建立、持續 open 的工作入口**，且已指派給 **`Sheng-wei-Tsai` 與 `Copilot`**。這表示主人傾向先把任務掛進 GitHub，再保留人機協作的可能性。
- 目前更像是**先建立追蹤容器、再補內容**的工作模式，而不是一開始就把需求寫完整。
- 龍蝦群對這個 repo 的長期記憶治理原則已相對明確：**在資訊不足時不臆測需求、不虛構功能方向、不把空白 issue 誤記成已定義任務**。
- 除了上述結構性事實外，**尚無足夠資料確認產品方向、技術棧、模組邊界、開發流程或穩定的業務規則**。

## Recent Themes
- 最近反覆出現的主題不是功能開發，而是**資訊缺口管理**。
- `#1` 一直被視為：
  - 已存在的入口
  - 但內容不足的佔位殼
  - 尚未進入可執行狀態的支線
- 多日快照一致指出，當前主要阻塞點不是程式實作，而是**需求定義尚未落盤**，包含：
  - 任務目標不明
  - 背景脈絡缺失
  - 使用情境未寫
  - 驗收標準未定
  - 外部討論尚未回填
- 近期也反覆浮現一個協作風險：**repo 外的對話（尤其 Telegram）可能握有真正背景，但尚未整理回 issue**，導致記憶鏈斷裂，代理與未來的主人都難以接手。
- 整體來看，這個 repo 最近比較像在**保留工作槽位**，而不是推進明確任務。若主人其實已有進展，目前也**沒有足夠證據顯示那些進展已被沉澱進 repo**。

## Constraints
- **資訊極度不足**：目前沒有足夠 issue 內容、留言、需求描述或決策紀錄可供可靠推論。
- **不可把 `#1` 視為已定義任務**：它目前仍比較像佔位入口，而非可直接拆解執行的工作單。
- **不可腦補 repo 外脈絡**：若 Telegram 或其他來源有討論，在未回填前都只能標記為可能存在，不能當成既定事實。
- **缺乏可驗證的完成條件**：目前無法判定什麼叫做完成、卡住、延期或已部分實作。
- **缺乏跨 issue 網絡**：近況幾乎只有 `#1`，因此無法建立可靠的優先序、依賴關係或產品地圖。
- **不確定性需明講**：目前甚至無法確定 `#1` 屬於產品需求、工程任務、流程整理，還是生活／工作交界的待辦。

## Open Loops
- `#1` 需要補上**真正的標題與任務定位**：這隻龍蝦到底要替主人處理什麼，現在仍看不清。
- 需要補齊 `#1` 的核心內容：
  - 背景來源
  - 問題陳述
  - 使用情境
  - 預期產出
  - 驗收標準
- 需要確認是否存在**repo 外對話紀錄**（如 Telegram），並整理回 issue，否則後續記憶仍會空轉。
- 需要釐清 `#1` 的**責任邊界與協作方式**：主人親做、Copilot 協作，還是只是先暫存想法。
- 需要確認是否有隱含但未寫出的：
  - 優先順序
  - 截止時間
  - 依賴項
  - 已做但未記錄的進度
- 若短期內仍不補資料，`#1` 會持續是一個**開著但不可執行的空殼入口**；這是目前最明確、也尚未解開的迴圈。
