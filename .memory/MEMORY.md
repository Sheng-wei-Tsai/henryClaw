# Repository Memory

## Stable Context
- 🦞 這個 repository 目前可確認的長期脈絡非常少；近一段時間的 issue 記錄幾乎只剩 **`#1` 這個已關閉的占位／測試性節點**，而且沒有需求描述、討論、實作紀錄或驗收條件。
- 🦞 龍蝦群已形成穩定共識：**沒有足夠上下文的 issue，不應被當成可延續的工程任務**。若卡片只是空殼，就應視為歷史痕跡，而不是活躍工作流。
- 🦞 目前這個 repo 的可用長期記憶，偏向「**維持追蹤品質與記憶衛生**」而非累積具體產品／技術知識；換句話說，龍蝦們現在更確定哪些東西**不是任務**，而不是哪些功能正在推進。
- 🦞 對 `#1` 的長期定位已相對明確：它是**已於 2026-05-15 關閉的無實質內容 issue**，不應再作為需求來源、開發依據或進度判讀基礎。

## Recent Themes
- 🦞 最近反覆出現的主題不是功能開發，而是**清理空白追蹤項目、降低雜訊**。
- 🦞 龍蝦群持續在做的事情比較像：
  - 確認哪些 issue 其實沒有形成任務
  - 避免把空白卡片誤認為仍在進行中的工作
  - 保留最小歷史紀錄，但不過度延伸解讀
- 🦞 多日 snapshot 都指出同一件事：**資料不足時要保守，不虛構進展**。目前沒有可靠證據顯示 repo 近期有明確的功能推進、除錯、規格收斂或跨 issue 協作。
- 🦞 若主人近期確實有做其他工作，那些脈絡**沒有出現在目前可讀的 issue 記憶中**；因此這份記憶只能保守記下「追蹤系統被整理過」，不能假裝知道更多。

## Constraints
- 🦞 **資訊來源受限**：目前 memory 可見內容極少，且幾乎都圍繞同一個空白 issue；因此很多 repo 背景、產品方向與技術決策都處於未知狀態。
- 🦞 **不可從空白 issue 反推需求**：缺少背景、目標、範圍、討論與驗收條件時，龍蝦群應避免自行補完故事。
- 🦞 **issue 可承接性偏低**：若任務建立時沒有留下基本上下文，後續 agent 或人類幾乎無法接手，只能重開新案或重新定義。
- 🦞 **外部記憶鏈可能缺失**：多份 snapshot 都提到，若曾有 Telegram 或其他地方的討論，目前並未被納入摘要；因此 repo 內記憶可能是不完整的。
- 🦞 **不確定性高**：現有資料不足以判定 `#1` 原本屬於產品、工程、流程管理，還是單純測試用途；這點應持續標記為未知，而不是寫成既定事實。

## Open Loops
- 🦞 **`#1` 若要重啟，必須先補齊基本定義**：
  - 明確目標
  - 背景脈絡
  - 範圍
  - 預期輸出
  - 驗收條件
- 🦞 需要確認：**主人是否有其他未被 issue 化、或未被摘要收進 memory 的實際工作脈絡**。目前看起來很可能有記錄缺口，但沒有證據能補上。
- 🦞 若未來還要把 issue 當成長期記憶來源，應建立更穩定的最小欄位習慣；至少要讓每張卡片能回答「這是什麼、為何存在、完成算什麼」。
- 🦞 目前沒有其他可明確列出的工程 open loop；這不是因為事情都完成了，而是因為**可用記錄太少，無法可靠辨識未完成事項**。
