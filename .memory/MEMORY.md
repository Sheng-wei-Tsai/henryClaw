# Repository Memory

## Stable Context
- 龍蝦群目前能穩定確認的 repo 現況很少：**近一段時間可見的工作脈絡幾乎只圍繞單一 open issue `#1`**，而且它長期處於「已建立入口、但內容未定義」的狀態。
- `#1` 已被指派給**主人與 Copilot**，可視為主人預設的人機協作入口；但這目前只代表協作意圖存在，**不代表任務已可執行**。
- 從多日快照反覆出現的模式來看，這個 repo 至少暫時呈現出一種工作習慣：**先建立 issue 作為追蹤容器，再視情況補內容**。不過這是否為主人長期穩定流程，資料仍偏少，需保留不確定性。
- 目前**沒有足夠證據**支持任何更具體的產品方向、技術主題、模組邊界、里程碑規劃或實作進展。龍蝦群只能保守記住：這裡現在比較像一個待整理的記憶殼，而不是已展開開發的工程現場。

## Recent Themes
- 最近最反覆浮出的主題是：**需求定義不足本身就是主要阻塞**。問題不在程式碼，而在缺少任務目標、背景脈絡、使用情境、限制條件與完成標準。
- `#1` 持續被當作**保留入口／佔位殼**使用：先讓工作有地方落腳，但尚未灌入足夠資訊讓龍蝦群或未來的主人自己接手。
- 多日摘要都提到**repo 外部脈絡可能存在但尚未回填**，尤其是 Telegram 或其他對話來源。這表示近期的核心工作不是實作，而是把散落在外部的需求與決策重新接回 issue 記憶鏈。
- 近期沒有看到可驗證的功能推進、bug 修復、設計決策或跨 issue 關聯圖。若主人有在其他地方推進事情，**目前尚未沉澱進這個 repo 的可追蹤記憶**。

## Constraints
- **資訊極度不足**：目前唯一可見的 issue 缺少標題語意、內文、留言與驗收標準，導致任何進一步推論都容易污染記憶。
- **不可腦補任務內容**：多日整理都一致指向同一條規則——在沒有正式需求與上下文前，不能把 `#1` 當成已定義任務，也不能替主人預設優先順序或解法。
- **外部對話未回填是協作風險**：若需求曾在 Telegram 或其他地方討論，未整理回 repo，就會讓後續 agent 無法可靠接手。
- **缺乏交叉驗證資料**：近 30 天幾乎沒有其他 issue 可互相印證，因此目前無法建立更高層的專案脈絡。
- **不確定性需明示**：現階段無法判定 `#1` 屬於工作專案、個人生活管理、流程整理，或只是暫存想法；也無法確認是否已有隱含截止時間、依賴項或實際進度。

## Open Loops
- `#1` 需要補上最基本的任務定義：
  - 要解決什麼問題
  - 服務哪個情境或對象
  - 預期產出是什麼
- `#1` 需要補齊可執行資訊：
  - 背景脈絡
  - 限制條件
  - 驗收標準
  - 優先順序
- 需要確認 repo 外是否已有相關討論，並回填到 issue：
  - Telegram 對話
  - 口頭決定
  - 其他筆記或需求來源
- 需要釐清責任邊界：這是主人親自處理、與 Copilot 協作、還是單純暫存的想法容器。
- 若短期內仍無法補齊內容，需再判斷 `#1` 是否應：
  - 繼續保留為入口
  - 重新命名／重寫內容
  - 或關閉以避免長期空殼化
