# Repository Memory

## Stable Context
- 龍蝦群目前能穩定確認的 repo 脈絡很少；這個記憶洞穴仍處於**早期建置階段**，可見資訊主要圍繞在「先建立可承接後續工作的記憶入口」。
- `#1` 已被當作一個**協作掛點／記憶容器**使用，並指派給主人與 Copilot。它目前比較像工作記憶的錨點，而不是已有明確範疇的實作任務。
- 從現有紀錄可推定，主人偏好把協作內容整理成**可持續接手的濃縮記憶**，而不是保存逐字聊天紀錄。這是目前最接近長期規則的工作方式訊號。
- 目前**沒有足夠證據**支持任何穩定的產品方向、技術棧決策、開發流程細節或領域知識主線；若硬寫成既定事實，會超出現有記憶所能承載的範圍。

## Recent Themes
- 最近數日的主旋律不是功能推進，而是**資訊稀薄與上下文缺口**：多份 daily snapshot 都顯示整理視窗內幾乎沒有可用 issue 內容。
- 在少數有內容的日期中，反覆出現的主題是：
  - **記憶基礎設施先行**：先有容器，再等內容補齊。
  - **阻塞點在需求定義，不在技術實作**：缺的是目標、背景、成功條件與角色分工。
  - **協作可讀性被重視**：希望後續脈絡能被整理成工程上可交接的摘要，而非散落對話。
- 2026-04-25 到 2026-04-29 的快照一致顯示：**沒有新的 issue 活動可供整理**。因此近期主題偏向「等待新資訊進場」，而不是「已有新方向形成」。
- 目前看不出穩定的跨 issue 主線；更準確地說，是**資料不足，尚無法辨識主線**。

## Constraints
- 目前最大的限制不是程式碼或架構，而是**原始資料不足**：缺少 issue 內容、需求背景、對話紀錄與驗收條件。
- 龍蝦群的記憶整理高度依賴 GitHub issue / comment 等可追溯來源；若沒有這些素材，就只能保守記錄，不能替主人腦補決策。
- `#1` 雖然存在，但目前仍偏向占位用途；在內容補齊前，不適合把它視為已定義完成的工作流或產品任務。
- shared manual notes 只提供了**記憶維護原則**，沒有額外的人工長期事實可供吸收；因此本份 MEMORY.md 的穩定內容仍然有限。
- 存在明確不確定性：目前無法判定主人是否在 repo 外有其他進展，只能說**現有快照沒有留下足夠痕跡**。

## Open Loops
- `#1` 需要補齊最核心的任務資訊：
  - 目標是什麼
  - 背景是什麼
  - 預期成果／輸出是什麼
  - 成功條件如何判定
- 需要釐清 `#1` 中各角色的責任分工：主人、Copilot、以及龍蝦 agent 分別負責開發、整理、追蹤，還是單純記憶承接。
- 需要補入可追溯的上下文來源，例如 issue 描述、留言、相關對話摘要或需求說明，否則後續記憶無法穩定延續。
- 需要在有新 issue 活動後重新整理 repo 主線；目前連「主要在做什麼」都還不能可靠定義。
- 近期持續觀察點：等待下一輪 issue 更新，確認這個 repo 會不會從「記憶容器」進一步長成有明確任務與決策的工作場域。
