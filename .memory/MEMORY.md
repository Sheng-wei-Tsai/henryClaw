# Repository Memory

## Stable Context
- 龍蝦群目前看到的 repo 工作脈絡非常稀薄；近 30 天內幾乎沒有可供長期歸納的 issue 流水，代表這個 repository 的公開追蹤面板暫時**不是高密度運作中的任務來源**。
- 目前唯一可確認的穩定協作傾向是：**資訊不足時不硬推任務**。若 issue 缺少背景、需求、目標或驗收條件，主人與龍蝦群傾向先停止延伸，而不是自行腦補成工程工作。
- 另一個可提煉的長期脈絡是：**重視 issue 衛生與追蹤品質**。占位、測試性、沒有內容承載力的紀錄，應盡早辨識並收束，避免污染後續記憶與優先序判斷。
- 若未來有真實需求，較合理的做法是：**另開新 issue 並補齊脈絡**，而不是回頭復用已關閉且資訊不足的舊殼。
- 不確定性：目前資料量太少，還無法判定 repo 的技術棧、產品方向、主要模組、開發節奏或固定工作流；這些都不能被寫成穩定事實。

## Recent Themes
- 最近最明確的主題不是功能開發，而是**清理追蹤系統**。唯一被看見的支線是 `#1`，且已被判定為占位／測試性紀錄並關閉。
- 這次整理反覆指向同一件事：**沒有內容的 issue 不值得持續佔用注意力**。主人近期像是在先把空殼任務撈出來確認，再把它們從工作池移除。
- 近期也浮現一個流程訊號：若主人希望龍蝦群未來能更好接手記憶與協作，**issue 描述品質需要更完整**，至少要能看出背景、目標與完成標準。
- 另一個近期觀察是：**目前看不到可執行中的主線任務**。這不一定代表主人沒有在做事，更可能代表工作尚未進入 issue 系統，或記錄密度不足以讓龍蝦群建立長期脈絡。

## Constraints
- **資料來源限制明顯**：目前 memory 幾乎只反映 issue 面向，而且近 30 天可用內容極少，因此很多判斷只能停留在流程層級，不能推導產品或實作細節。
- **不可從空白 issue 推導需求**：像 `#1` 這類沒有描述、沒有討論、沒有上下文的紀錄，不應被延伸成規格、功能或待辦。
- **目前沒有足夠證據建立 repo 慣例**：例如分支策略、PR 規範、測試要求、部署流程、標籤系統等，都尚未出現在記憶中。
- **標籤資訊缺失**：目前沒有可用 labels，可見 issue 分類機制尚未形成，或至少尚未反映在這批記憶裡。
- **不確定性需保留**：由於多日快照都顯示「沒有可用 issue」，任何關於主人近期專案狀態的結論都應保守處理。

## Open Loops
- 是否需要建立或強化 **issue 模板**，避免再次出現只有標題、沒有內容的追蹤殼。
- 主人是否其實有正在進行中的工作，但**尚未被記錄到 issue 系統**；若有，龍蝦群目前還掛不上記憶網。
- 若 `#1` 背後原本有真實想法，現在仍缺少三個核心資訊：**需求內容、目標、完成標準**。
- repo 的實際用途、技術方向與主要工作流仍然**高度不明**；需要未來更多有內容的 issue 或人工筆記，才能從「追蹤清理」進一步長成真正的長期記憶。
