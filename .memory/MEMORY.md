# Repository Memory

## Stable Context
- 龍蝦群目前能確認的 repo 長期脈絡非常薄，**沒有足夠證據**顯示這裡已有穩定的產品主線、技術方向、維運節奏或多條並行中的工作流。
- 近一段時間內，唯一反覆被確認的 issue 是 **#1**；它已被一致判定為**占位／測試性／空白節點**，不是實際展開過的任務主線。
- 對這個 repo 而言，龍蝦們已形成一條可重用的記憶規則：**沒有需求背景、目標、討論脈絡與驗收條件的 issue，不應被視為有效待辦。**
- 目前記憶系統採取**保守整理策略**：資料不足時，優先標記「無法判定」與「不主動延伸」，避免替主人腦補不存在的進度或任務。
- `#1` 的關閉狀態可視為**有效結案**，不是暫停中的工作；若未來要延續相關想法，較合理的做法是補齊脈絡後重啟，或直接建立新 issue。

## Recent Themes
- 最近幾天的主題幾乎都不是功能開發，而是**記憶清點、任務去雜訊、issue hygiene**。龍蝦們像是在替主人清理抽屜，把空白便條從「可能待辦」改標成「只是殼」。
- 多日反覆出現的訊號是：**#1 沒有實質內容，也沒有後續對話、實作或驗證成果**，因此不再值得持續追蹤。
- 近期整理工作強調一件事：**需求明確性比任務數量更重要**。若 issue 缺少目標、背景與成功條件，就無法形成可靠的 agent 工作鏈。
- 另一個反覆浮現的主題是**外部脈絡缺席**：目前看不到 Telegram、comment、commit、PR 或其他來源能補足 `#1` 的上下文，因此無法還原更多真實工作進展。
- 整體來看，最近的 repo 活動更像是**確認哪些事情其實沒有開始**，而不是推進某個已知專案。

## Constraints
- **資料密度極低**：近 30 天可整理出的內容幾乎只有 `#1`，且內容近乎空白，導致無法可靠推導更高階的 repo context。
- **不可腦補**：當 issue 缺少需求、背景、討論與驗收資訊時，龍蝦們應維持保守，不自行補做規劃、拆解或進度推論。
- **記憶來源可能不完整**：若主人有在 Telegram、聊天紀錄、commit、PR 或 repo 外工具中討論工作，這些脈絡目前沒有進入記憶池，因此 repo 內觀測結果可能失真或不完整。
- **空白 issue 不應消耗追蹤成本**：像 `#1` 這類沒有任務內容的節點，應視為無法執行，而不是掛著等待龍蝦自行理解。
- **目前不存在可驗證的穩定工程規範**：除了「需求不足就不延伸」這條記憶紀律外，尚無足夠資訊支持更多長期流程規則。

## Open Loops

  - 為什麼要做
  - 想解決什麼問題

- **Issue 建立流程是否需要更明確模板**是個合理但尚未定案的問題；近期反覆暴露出「有 issue、但沒有任務定義」的情況。
- **是否有其他未被 issue 化的工作主線**目前無法判定。現有資料不足，不能據此斷言主人最近完全沒有其他工程或生活／工作事項。
- 整體不確定性仍高：這份記憶更像是在替主人守住一條底線——**目前沒有足夠證據支持更多長期結論**。
