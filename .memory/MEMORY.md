# Repository Memory

## Stable Context
- 🦞 這個 repository 目前可確認的長期上下文非常稀薄；近一段時間內，龍蝦群幾乎只看到一筆可辨識的 issue：`#1`。
- 🦞 `#1` 已在 `2026-05-15` 關閉，且多日摘要一致指出它屬於**空白占位／測試型 issue**，沒有形成真正的需求、討論、實作或驗收脈絡。
- 🦞 因此，對這個 repo 目前最穩定的認知不是「正在推進某項功能」，而是：**現有 issue 記錄不足以承載可靠的工程記憶**。
- 🦞 龍蝦群已形成一個穩定工作原則：**沒有明確上下文，就不把 issue 包裝成真實進展**。這比補腦更重要，因為長期記憶一旦污染，後續協作會持續偏航。
- 🦞 另一個可重用的 repo context 是：**若某個想法沒有被寫成目標、背景、範圍與驗收條件，它就不應被視為可承接任務**。目前 `#1` 正是這種反例。

## Recent Themes
- 🦞 最近反覆出現的主題不是功能開發，而是**追蹤系統的資訊品質與議題衛生整理**。
- 🦞 多日 snapshot 都在做同一件事：確認 `#1` 不具延續價值，避免它繼續佔據注意力或被誤認為未完成工作。
- 🦞 近期可觀察到的節奏比較像：
  - 清理無效 issue
  - 校正「哪些其實不是工作」
  - 保守標記資料不足，而不是虛構產品／工程敘事
- 🦞 另一個近期反覆浮現的訊號是：**daily memory 的可用性高度依賴原始 issue 品質**。當 issue 缺少描述、留言或外部對話摘要時，龍蝦群只能記下「這裡有殼，但沒有足夠內容可延續」。
- 🦞 目前沒有足夠證據顯示 repo 近期存在穩定推進中的功能線、除錯線、文件線或生活任務線；若有，至少尚未在這批記憶來源中留下可驗證痕跡。

## Constraints
- 🦞 **不可憑空補完需求。** 對於像 `#1` 這種沒有背景與內容的 issue，應明確標記為資訊不足，而不是推測其原意。
- 🦞 **GitHub issue / comment 才是主要原始來源。** 若真正脈絡存在於 Telegram 或其他外部對話，但沒有回填到 repo，可追溯性就會中斷。
- 🦞 **目前資料量不足。** 近 30 天摘要幾乎沒有提供可支撐技術決策、產品方向或工作節奏判讀的材料，因此任何較強結論都應保留。
- 🦞 **空白 issue 不適合作為後續工作的承接點。** 若未來要重啟相關想法，應重新建立有內容的新 issue，或先補齊原始背景。
- 🦞 不確定性需直接寫明：目前無法判定 repo 是否真的沒有其他工作，只能說**在現有 memory 來源中看不到**。

## Open Loops
- 🦞 若主人當初在 `#1` 背後其實有真實想法，那條線目前仍是未解的；缺的至少包括：
  - 目標
  - 背景脈絡
  - 範圍界定
  - 驗收條件
- 🦞 多日摘要都提到同一個潛在缺口：**是否有未匯入的外部對話（例如 Telegram）其實對應 `#1`**。目前無法確認，這是主要不確定來源。
- 🦞 若主人未來希望這個 repo 的 memory 系統能真正支援協作，後續需要改善 issue 建立品質，至少補齊：
  - 為何要做
  - 要做到什麼
  - 怎樣算完成
- 🦞 除上述之外，現階段**沒有可明確列出的工程 open loop**；這不是因為事情都完成了，而是因為可驗證記錄不足。
