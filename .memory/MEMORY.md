# Repository Memory

## Stable Context
- 龍蝦群目前能穩定確認的 repo 脈絡非常少；這個記憶洞穴在近一段時間內，**幾乎只有一條可見工作線 `#1`**。
- 主人似乎有一種工作習慣：**先開 issue 當容器，再視情況補內容**。這不是一次性的單日現象，而是在多天快照中反覆出現的模式。
- `#1` 長期呈現同一種狀態：**issue 已建立、保持開啟、且曾指派給主人與 Copilot**，但內容沒有長成可執行任務。
- 因此，對這個 repo 來說，目前比較像是**記憶基礎設施已存在，但實際知識尚未灌入**；issue 系統被當作追蹤入口，而不是已成熟的任務規格來源。
- 目前**沒有足夠證據**顯示此 repo 已形成穩定的產品方向、技術棧決策、開發流程、標籤規則或里程碑節奏。

## Recent Themes
- 近期反覆出現的核心主題只有一個：**任務定義不足**。
- 多日快照都指出 `#1` 是一個「先佔位、後補充」的空白工作節點；龍蝦們能做的主要不是推進實作，而是反覆提醒：**這裡有容器，但沒有任務本體**。
- 另一個持續浮現的訊號是：**人機協作意圖存在，但分工與目標缺席**。主人與 Copilot 被放進同一條 issue，表示協作框架先被架起來，但沒有誰定義、誰執行、誰驗收的說明。
- 近期也持續暴露出**外部脈絡未回寫**的問題：若任務原本來自 Telegram、口頭討論或其他筆記，目前都沒有被整理回 repo，因此後續 agent 幾乎無法接手。
- 整體來看，最近這個 repo 的主旋律不是開發進展，而是：**避免把空白 issue 誤記成已有明確規劃的工作**。

## Constraints
- **資訊極度不足**：目前可見資料幾乎只圍繞 `#1`，且內容空白，不能可靠推論任務性質。
- **不可過度腦補**：在沒有標題語意、正文、留言或外部對話佐證時，龍蝦群應把 `#1` 視為待定義容器，而不是功能、研究、生活提醒或 bug。
- **可接手性低**：缺少需求來源、背景、成功條件、優先級與完成標準，任何 agent 都難以安全接手。
- **外部知識未匯入 repo**：若主人有在 Telegram 或其他地方討論，現階段記憶系統看不到；這是 repo 記憶品質的主要限制。
- **缺乏穩定規則樣本**：目前沒有足夠歷史資料可提煉出更高信心的 repo 慣例，因此許多判斷都必須保留不確定性。

## Open Loops
- `#1` 到底要承接什麼主題，仍未明：是功能、文件、研究、流程優化、生活事項，還是其他類型任務？
- `#1` 需要補上最基本的任務定義：
  - 明確標題
  - 背景來源
  - 預期成果
  - 成功標準
  - 可執行範圍
- 需要確認是否存在**尚未回寫的外部脈絡**，例如 Telegram 對話、口頭決定或其他筆記；若有，應整理進 issue。
- 需要釐清主人與 Copilot 在 `#1` 的角色分工：誰主導定義、誰負責執行、誰負責驗收，目前完全不明。
- 需要評估 `#1` 是否應持續保持 open：
  - 若只是暫存坑位，應補內容或重新命名；
  - 若其實沒有實際待辦，可能要考慮關閉，避免長期佔位造成記憶污染。
- 目前也無法確認 repo 是否還有其他未被 issue 化的重要工作線；若有，龍蝦群現在**看不到，也無法替主人穩定保存**。
