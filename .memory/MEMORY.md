# Repository Memory

## Stable Context
- 龍蝦群目前能確認的長期脈絡非常少；這個 repository 在最近可見記憶中，**沒有穩定展開中的功能線、產品主題或工程主軸**。
- 目前唯一反覆出現且可視為穩定規則的訊號是：**空白或缺乏上下文的 issue 不應被當成有效工作入口**。若任務沒有目標、背景、範圍或預期產出，龍蝦們不會把它延伸成實作需求。
- 已知的 `#1` 被一致判定為**無實質內容的占位 issue**，且已關閉；它**不應再作為後續工作的依據**。若主人未來想重啟相關方向，應重新建立新 issue，而不是沿用舊殼。
- 從現有資料看，這個 repo 的近期記憶更像是**待辦治理與資訊清理**，而不是持續中的開發紀錄。

## Recent Themes
- 最近多日快照反覆指向同一件事：**沒有新的 issue 活動可整理**，龍蝦群多半只能保留既有記憶。
- 在有內容的幾天裡，主題集中在**清理沒有形成需求的占位任務**，避免 backlog 被空白卡片污染。
- 反覆浮現的工程管理訊號：
  - **待辦可信度比表面數量更重要**：與其保留空白 issue，不如直接關閉。
  - **記錄品質會直接影響後續接手能力**：沒有描述、沒有討論、沒有來源脈絡時，龍蝦群無法可靠還原主人原意。
  - **目前資料密度偏低**：看不出近期有穩定推進中的功能、修 bug、文件補強或流程自動化主題。
- 不確定性：這不一定代表主人真的沒有其他工作進展，也可能只是**那些進展尚未進入 issue 記錄**，因此記憶池無法觀測。

## Constraints
- **資料來源限制明顯**：目前 memory 幾乎只反映 issue 摘要；若工作沒有被 issue 化，龍蝦群就難以保留可重用脈絡。
- **不可從空白 issue 推導需求**：缺少目標、背景、預期成果、相關討論或外部連結時，不應自行腦補任務內容。
- **現有資訊不足**：最近多天都沒有可用 issue，因此任何關於 repo 長期方向、技術棧偏好、協作節奏或生活工作習慣的判斷，都應保持保守。
- **記憶品質依賴輸入品質**：若主人希望未來快照更有用，issue 至少需要包含：
  - 任務目的
  - 背景脈絡
  - 範圍或預期產出
  - 必要時附上外部討論摘要或連結

## Open Loops
- **`#1` 的原始意圖仍不可考**：若主人其實記得當初想做什麼，需要另開新 issue 並補齊上下文。
- **repo 是否存在未被記錄的實際工作線**：目前無法判斷；這是資料缺口，不是已確認的「沒有工作」。
- **後續是否會恢復可整理的 issue 活動**：最近多日快照都在等待下一輪更新，龍蝦群暫時只能守著空殼，不敢假裝有新進展。
- **長期規則仍待建立**：除了「不要依賴空白 issue」之外，現階段還沒有足夠證據整理出更完整的 repo 慣例、技術決策或主人工作節奏。
