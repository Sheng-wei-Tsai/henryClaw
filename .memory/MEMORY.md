# Repository Memory

## Stable Context
- 龍蝦群目前能確認的長期脈絡很少；這個 repo 在近一段時間內**沒有穩定可辨識的產品主線、功能開發節奏或維運議題**。
- 唯一反覆被確認的穩定事實是：**Issue #1 已被視為占位／測試性紀錄，而非真正展開過的工作項目**。它已於 `2026-05-15` 關閉，且缺乏需求描述、背景、討論與驗收條件。
- 這個 repo 的記憶整理原則偏保守：**沒有足夠上下文時，不腦補需求、不虛構進度、不把空白 issue 當成有效待辦**。
- 對龍蝦群來說，這裡目前更像是一個**需要維持記憶衛生的空白工作面**，而不是已有明確長期工程脈絡的活躍專案。

## Recent Themes
- 近期反覆出現的主題不是功能推進，而是**記憶清點、任務去雜訊、issue hygiene**。
- 多日快照都在做同一件事：反覆確認 **#1 沒有實質內容，應停止追蹤**，避免未來回顧時把它誤認成漏掉的工程工作。
- 另一個近期主題是：**資料不足本身需要被明確記錄**。目前沒有足夠證據顯示 repo 內有新的開發、修 bug、規格討論、部署工作，或與生活事件交織的工作脈絡。
- 龍蝦們近期的主要價值在於**守住記錄可信度**：寧可誠實寫「不知道／沒有資料」，也不製造看似完整但其實虛假的任務敘事。

## Constraints
- **原始事實來源仍以 GitHub issue/comment 為主**；若只有模糊標題或空白內容，不能視為可執行需求。
- **缺少目標、背景、預期成果或驗收條件的 issue，不應被當成有效 backlog**。
- 若相關脈絡其實存在於 **Telegram** 或其他未匯入來源，龍蝦群目前**無法自行還原**；未補齊前只能維持「資料不足」判定。
- 目前資訊量非常有限，因此：
  - 無法可靠推斷 repo 的真正專案主線。
  - 無法確認是否有被 commit、PR、外部對話承接的隱含工作。
  - 無法把單日空白訊號升格為穩定工作規則以外的更多結論。
- 共享手動筆記的角色是保留**長期規則、決策與限制**，不是複製 issue 原文或充當日誌索引。

## Open Loops
- **#1 若未來要重啟或重用，需要主人先補齊：**
  - 明確目標
  - 背景脈絡
  - 預期產出
  - 驗收條件或完成定義
- 若曾有 **Telegram 或其他外部討論**支撐 #1，仍待匯入；在補齊之前，龍蝦群會繼續把它視為**已關閉且無實質需求的空白節點**。
- 目前最大的未解環不是某個功能卡住，而是**整體 repo 缺乏足夠可追蹤上下文**；是否存在真正的後續工作，現有記憶無法確認。
