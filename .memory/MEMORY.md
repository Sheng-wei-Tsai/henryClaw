# Repository Memory

## Stable Context
- 龍蝦群目前能確認的長期穩定事實很少；這個 repo 的可用記憶仍相當稀薄，**不能假設已有明確產品方向、技術棧慣例或固定工作流**。
- 目前唯一可穩定記住的協作脈絡是：主人有在使用 **GitHub Issue 作為工作追蹤容器**，至少已建立一條支線 `#1`。
- 龍蝦群已形成一條保守整理原則：**當 issue 缺乏標題、背景、需求與完成條件時，不把它視為已定義完成的任務，也不自行腦補技術方向。**
- 目前沒有足夠證據支持以下任何穩定結論：  
  - repo 是否聚焦某個特定產品或功能  
  - 是否與 Telegram 有穩定關聯  
  - 是否已有固定的人機分工模式  
  - 是否存在既定開發節奏或驗收流程

## Recent Themes
- **先佔位、後補需求**  
  近兩天反覆出現的主題是：主人似乎先把一件事放進 issue 系統中，避免工作線完全散失，但尚未把內容整理成可執行任務。
- **協作入口已存在，但可接手性不足**  
  `#1` 已經是可追蹤節點，但因為缺少描述、背景與成功條件，其他龍蝦或未來的主人自己都難以順利接手。
- **資訊不足被明確視為現況，而非可忽略細節**  
  龍蝦群近期一致採取保守判斷：承認不知道，比錯誤補完更重要。現階段沒有看到可確認的產品決策、技術決策或實作進展。
- **可能存在人機協作意圖，但尚未成形**  
  2026-03-26 的快照提到 `#1` 有主人與 Copilot 的指派訊號；這可視為近期線索，但因缺乏更多樣本，**還不能升格為穩定規則**。

## Constraints
- **原始資訊極少**：近 30 天記憶中實際只看到 `#1`，且內容近乎空白，導致大部分判斷都必須保守處理。
- **禁止過度推定**：在主人補齊資訊前，不應自行推論任務屬性、功能範圍、技術方案或外部來源。
- **issue 目前不具備工程落地所需最小資訊**：缺少目標、背景、完成條件、優先順序、進度與阻礙。
- **知識交接風險高**：若工作只停留在空殼 issue，後續協作者難以判斷該做什麼，也難以驗證何時算完成。
- **存在不確定性**：  
  - `#1` 是否真為正式任務，或只是臨時佔位，尚不明確。  
  - 指派資訊是否代表固定協作模式，也尚不明確。  

## Open Loops
- **補完 `#1` 的基本定義**
  - 改成有意義的標題
  - 補上任務描述與背景
- **說明需求來源**
  - 這條支線是來自生活事件、臨時想法、外部需求、bug，還是其他脈絡，目前完全不清楚。
- **定義完成條件**
  - 需要明確寫下什麼算完成、預期產出是什麼、是否有驗收方式。
- **補上目前進度與阻礙**
  - 若主人其實已經做了一部分，現有記憶完全無法反映。
- **釐清協作分工**
  - 若 `#1` 確實預期由主人與 AI／Copilot 協作，需補上誰負責定義、誰負責實作、誰負責驗收。
- **等待更多樣本形成穩定 repo context**
  - 目前資料不足，還無法整理出更高可信度的長期規則；後續若有更多 issue、留言或決策紀錄，應再重新蒸餾。
