# Repository Memory

## Stable Context
- 龍蝦群目前能確認的穩定背景很少：這個 repo 的長期記憶流程依賴 **GitHub issue / comment 作為原始事實來源**，記憶文件只應做整理與蒸餾，不應逐段複製原文。
- `.memory/shared/manual.md` 是給人類手動維護的長期記憶區，適合放：
  - 穩定規則
  - 長期決策
  - 常見限制
  - agent 共同遵守的 repo 習慣
- compact-memory workflow 會讀取手動筆記，但**不會覆寫**它；因此手動筆記可視為較穩定、可持續累積的記憶層。
- 目前從最近多日快照中，**看不出任何穩定的專案主題、工作模式或功能脈絡**。這不是代表 repo 沒有內容，而是代表在觀測視窗內沒有可用 issue 資料可供龍蝦群萃取。

## Recent Themes
- 最近一段時間（至少 2026-04-26 到 2026-05-02），每日快照都顯示：
  - 沒有可用 issue
  - 沒有可辨識的跨 issue 主題
  - 沒有新的跨 issue 決策
- 因此近期主題可歸納為：**記憶來源暫時中斷，整理流程處於等待新資料的空窗期**。
- 這段期間沒有足夠訊號可判斷：
  - 主人最近在做哪類工作
  - repo 是否有活躍開發
  - 是否存在新的長期規則或偏好

## Constraints
- 記憶整理必須以 **issue / comment** 為主要依據；若缺少這些來源，就不應憑空推定長期事實。
- MEMORY 類文件應是 **curated long-term memory**，不是原始日誌、不是 issue 索引，也不是每日摘要的逐條拼貼。
- 若某件事只在單日短暫出現，且尚未形成穩定模式，應放在近期主題或 open loop，而**不要寫成穩定規則**。
- 當資訊不足或彼此矛盾時，應明確標示不確定性。以目前資料量來看，**不確定性很高**，不宜過度推論 repo 背景。

## Open Loops
- 等待下一輪有實際內容的 issue / comment 出現，才能恢復有效的長期記憶整理。
- 需要未來資料來回答的關鍵未知：
  - 這個 repo 的主要用途與技術脈絡是什麼？
  - 主人目前關注的長期任務、維護習慣與工作節奏為何？
  - 是否已有隱含但尚未寫入 manual notes 的 repo 慣例？
- 若後續仍持續多日沒有 issue，可能需要確認：
  - 是 repo 真的處於低活動狀態，
  - 還是記憶擷取流程／資料來源設定有缺口。
