# Repository Memory

## Stable Context
- 龍蝦群目前能確認的長期脈絡非常少；這個 repository 在最近可見記憶中，**沒有穩定且持續推進的功能主線**，更像是一個尚未形成明確工作流的空白或早期倉庫。
- 目前唯一可辨識的 issue 是 **#1**，且已於 **2026-05-15 關閉**。多日快照一致指出它屬於**占位／測試性質**，沒有足夠需求、背景、討論或驗收資訊，因此**不應被視為有效的歷史任務脈絡**。
- 龍蝦群已反覆收斂出一條可當作 repo 慣例的記憶：**沒有明確目標、範圍與完成條件的 issue，不應持續佔用追蹤注意力**。若只是空殼紀錄，應關閉或以新 issue 重新定義。
- 目前看不到可靠證據顯示此 repo 已建立固定的技術方向、模組邊界、部署流程、協作節奏或標籤治理方式；在這些資訊出現前，龍蝦們只能把它記成**上下文不足的倉庫**，而不是成熟專案。

## Recent Themes
- 最近幾天的主題不是開發，而是**記憶清艙**：龍蝦們主要在確認哪些追蹤項其實沒有內容，避免主人之後誤把空白 issue 當成未完成工作。
- **#1 的結案**是近期唯一明確事件。它被多次確認為不承載實質需求，因此近期整理重點是「收束無效入口」，不是延伸功能。
- 多份 daily snapshot 都反覆指出一個訊號：**repo 外部可能存在脈絡斷裂**。若主人曾在 Telegram 或其他地方討論過 #1 背後的想法，目前這些內容沒有回流到 issue，因此龍蝦群無法重建完整記憶。
- 近期也浮現一個流程層面的主題：**issue 建立品質不足會直接削弱後續協作與記憶整理**。這不是已定案的制度，但已是連續多日出現的觀察。

## Constraints
- **資訊極度不足**：近 30 天可用 issue 幾乎只有 #1，且內容空泛，導致龍蝦群無法可靠推導產品方向、技術決策或待辦優先序。
- **外部上下文不可見**：若工作討論散落在 Telegram、聊天訊息或其他工具，而未回寫到 repo，龍蝦群就無法替主人保存可重用的工程記憶。
- **不可從空白 issue 過度推論**：目前已形成明確限制——在缺少需求描述、背景與驗收條件時，不應自行腦補成功能規格或實作任務。
- **現有資料可能失真於「未被 issue 化的工作」**：不能排除主人其實有在做事，只是沒有進入這批 issue 記錄；因此「目前沒有主線」是基於可見資料的結論，不等於真實世界完全沒有進展。
- **標籤與分類訊號缺失**：目前沒有 labels 可協助辨識任務類型、優先級或領域。

## Open Loops
- **是否需要建立或採用 issue 模板？**  
  近期反覆暴露出「有 issue 但沒有任務內容」的問題；若主人希望龍蝦群能長期接手記憶，至少需要背景、目標、範圍與驗收條件。
- **#1 背後是否其實有遺失的外部脈絡？**  
  若曾在 Telegram 或其他地方討論過，仍需補回摘要或連結；否則這條線應繼續視為已結束且不可復用。
- **未來若有相關想法，是否要另開新 issue？**  
  目前共識是：若要重啟任何與 #1 類似的念頭，應重新開案，不要沿用舊殼。
- **repo 是否存在未被記錄的實際工作？**  
  這點目前無法確認。若主人近期有開發、整理、部署或生活流程相關工作，龍蝦群還沒拿到足夠記憶碎片。
- **長期專案定位仍不明**：  
  目前無法判定 `henryClaw` 是測試倉庫、個人工具、正式專案雛形，還是單純用來驗證流程的空殼；這需要後續更多實際內容才能定錨。
