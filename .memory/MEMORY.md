# Repository Memory

## Stable Context
- 🦞 這個 repo 目前可確認的長期工程脈絡非常少；近 30 天的 issue 記憶中，唯一反覆被提到的是 **#1**，而它已被一致判定為**空白占位／測試性 issue**，並已於 **2026-05-15 關閉**。
- 🦞 龍蝦群已形成穩定共識：**#1 不應再被視為有效待辦、需求來源或可延續的工程上下文**。它沒有需求描述、沒有討論紀錄、沒有實作痕跡，也沒有驗收條件。
- 🦞 這個 repo 目前更像處於**記憶整理與追蹤品質校正**階段，而不是有明確功能線正在持續推進。至少從現有 memory 來源，無法可靠還原產品方向、技術棧決策或協作流程細節。
- 🦞 若未來有與 **#1** 類似的想法要重新啟動，較穩妥的做法是：**重新定義需求後另開新 issue，或先補齊背景再啟動**，不要直接依賴這張空白舊殼。

## Recent Themes
- 🦞 最近幾天反覆出現的主題不是開發進展，而是**清理無效追蹤項**：龍蝦們持續確認哪些 issue 其實沒有內容，避免把空白卡片誤認為仍在進行中的工作。
- 🦞 **資訊不足要明講，不補腦** 是近期最穩定的整理原則。對於缺少背景、目標、驗收條件的 issue，龍蝦群傾向保守歸檔，而不是虛構需求或進度。
- 🦞 **議題衛生與可承接性** 是近期唯一明顯的流程主題：若 issue 沒有基本上下文，就無法成為可靠的長期記憶節點，也無法讓後續 agent 或人類接手。
- 🦞 近期資料沒有顯示出新的功能開發、除錯、文件補強、架構決策或多線協作。若主人實際上有在推進其他工作，**目前 memory 中沒有足夠證據**。

## Constraints
- 🦞 **資料來源嚴重稀薄**：目前 memory 幾乎只反映到 #1，缺乏其他 issue、留言、提交脈絡或外部對話摘要，因此很多事情只能標記為未知，不能當成穩定事實。
- 🦞 **不可從空白 issue 推導需求**：沒有目標、範圍、背景、驗收條件的卡片，不應被當作工程任務依據。
- 🦞 **外部對話未回填會造成記憶斷鏈**：多份 snapshot 都提到，若原始脈絡來自 Telegram 或其他外部討論，但沒有摘要回寫到 repo 記憶，龍蝦群就無法可靠延續。
- 🦞 **目前沒有足夠證據建立 repo 的穩定技術規則**：例如架構偏好、部署方式、測試策略、分支慣例等，都未在現有資料中出現；這些欄位目前應保持空白，而不是猜測。

## Open Loops
- 🦞 **若要重啟 #1 或其背後想法**，仍缺少：
  - 明確目標
  - 背景脈絡
  - 範圍界定
  - 預期成果
  - 成功條件／驗收方式
- 🦞 **是否存在未被記錄的其他工作線** 仍不確定。多日 snapshot 都指出：若主人有推進其他事項，它們沒有出現在目前的 issue 記憶中。
- 🦞 **外部對話摘要是否存在** 仍不確定。若 #1 或其他工作原本有 Telegram／外部討論，相關摘要尚未進入 memory，這條記憶鏈目前仍是斷的。
- 🦞 後續若要讓這份長期記憶更有用，優先需要補的是：**有內容的 issue、討論摘要、決策紀錄與可驗收的任務描述**；否則龍蝦群只能繼續記住「這裡曾有一張空殼，但沒有長成工作」。
