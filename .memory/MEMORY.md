# Repository Memory

## Stable Context
- 龍蝦群目前能確認的穩定脈絡很少：這個 repo 的記憶整理流程會讀取 `.memory` 內的人工作業筆記與每日快照，目標是保留**長期可重用的脈絡**，而不是複製原始 issue 內容。
- `shared/manual.md` 是保留給人類手動維護的長期記憶區，適合放：
  - 穩定規則
  - 長期決策
  - 常見限制
  - agent 共同遵守的 repo 習慣
- 記憶來源的原始依據仍應以 **GitHub issue / comment** 為準；記憶文件只做整理與蒸餾，不應變成原始資料備份。
- 目前提供的 daily snapshots 在連續多日內都沒有可用 issue，因此**尚無足夠證據**推導出此 repo 的實際工作內容、技術棧、開發流程或主人的穩定偏好。

## Recent Themes
- 最近一段時間（至少 2026-05-05 到 2026-05-11），每日整理結果一致：
  - 沒有可用 issue
  - 沒有可辨識的跨 issue 主題
  - 沒有新的跨 issue 決策
- 因此近期主題不是「某項功能正在推進」，而是更偏向：
  - 記憶系統處於**低活動或無輸入**狀態
  - 龍蝦群目前只能維持既有記憶，不宜自行腦補 repo 背景
- 這也表示：若主人近期有在其他地方工作，這份記憶尚未接收到對應訊號。

## Constraints
- 不可把 daily snapshots 當成長期事實來源直接逐段複製；應優先蒸餾成可重用的 repo context。
- 不可把短暫、只出現在最新一天的訊息寫成穩定規則；若證據不足，應放在近期主題或 open loop。
- 若資訊不足或彼此矛盾，必須明確標示不確定性。以目前資料量來看，**不確定性很高**。
- `shared/manual.md` 會被 workflow 讀取，但不會被覆寫；這代表人類手動補充的長期記憶具有特殊地位，適合承載經確認的規則與決策。
- 目前沒有 issue 標籤、跨議題摘要或決策可供交叉驗證，因此任何關於專案方向、優先級、慣例的推論都應保守處理。

## Open Loops
- 等待下一輪有實際內容的 issue / comment 出現，才能建立更可靠的長期記憶。
- 需要未來補充的關鍵資訊包括：
  - repo 的主要用途與技術範圍
  - 主人或團隊的工作習慣
  - 穩定的開發規則、命名慣例、PR / issue 流程
  - 已做出的長期決策與不可違反的限制
- 若主人希望這份記憶更有用，最有效的做法是：
  - 在 `shared/manual.md` 手動補入已知穩定規則
  - 或讓 GitHub issues / comments 開始承載可追溯的工作脈絡
- 目前最大的未解環節：**這個 repo 缺乏可觀測活動**，龍蝦群只能先守著空殼記憶，等待下一批真正的訊號上岸。
