# Repository Memory

## Stable Context
- 🦞 這個 repository 目前可確認的長期脈絡非常稀薄；在近 30 天可見的 issue 記憶中，唯一反覆出現的是 **#1**，而它已被一致判定為**沒有實質需求的占位／測試型 issue**。
- 🦞 **#1 已於 2026-05-15 關閉**，且不應再被視為有效的工作入口、需求來源或開發依據。
- 🦞 龍蝦群對這個 repo 的共同整理原則已相當明確：**沒有需求描述、背景、討論脈絡與驗收條件的 issue，不應被延伸成工程任務**。
- 🦞 目前看起來，這個 repo 的 issue 記憶更像在做**追蹤系統衛生整理**，而不是承載穩定的產品開發脈絡；若未來要建立可持續的長期記憶，需要更完整的 issue 內容作為基底。
- 🦞 有些摘要多次提到外部對話來源（例如 Telegram）可能存在，但**目前沒有足夠證據證明其內容已匯入 repo 記憶**；因此這只能視為可能性，不能當成既定事實。

## Recent Themes
- 🦞 最近最明顯的主題不是功能推進，而是**記憶除草與追蹤去噪**：龍蝦群反覆確認哪些卡片其實沒有長成真正任務，避免空白節點污染後續判讀。
- 🦞 **#1 的收束** 是這段期間唯一穩定可見的事件：它被反覆確認為已結束、無內容、不可承接。
- 🦞 **資訊不足要明講，不補腦** 是近期反覆出現的工作態度。多份 daily snapshot 都強調：看不到需求、討論或實作，就不要虛構產品方向、技術決策或生活脈絡。
- 🦞 近期資料沒有顯示出明確的：
  - 功能開發
  - bug 修復
  - 規格收斂
  - 文件補強
  - 多 issue 並行協作
- 🦞 因此，最近這個 repo 給出的訊號比較像：**主人正在整理追蹤系統邊界，但尚未留下足夠可重用的工程上下文**。

## Constraints
- 🦞 **資料來源限制很強**：目前 memory 幾乎只看得到單一 issue（#1），而且內容空白，導致龍蝦群無法可靠重建更大的專案脈絡。
- 🦞 **不可從空白 issue 推導需求**。若 issue 缺少目標、背景、範圍、討論與驗收條件，就只能標記為資訊不足，不能當作待辦。
- 🦞 **關閉狀態在此 repo 具有強烈歸檔意義**：像 #1 這樣已關閉且無補充內容的卡片，應視為歷史殘留，而非暫停中的工作。
- 🦞 **外部對話未匯流時，記憶會斷裂**。若真有 Telegram 或其他外部討論，但沒有回填摘要，agents 就無法延續上下文。
- 🦞 **目前不確定 repo 是否真的沒有其他工作，只能確定現有記錄不足**。也就是說，限制不一定來自主人沒做事，而可能來自追蹤資料沒有被寫進 issue 記憶。

## Open Loops
- 🦞 **若主人想重啟 #1 背後的想法**，需要重新建立可執行上下文；目前至少缺少：
  - 明確目標
  - 背景脈絡
  - 範圍界定
  - 預期輸出
  - 驗收條件
- 🦞 **應優先另開新 issue，而不是依賴 #1**。因為 #1 已被長期視為空白占位卡，沿用它容易再次造成混淆。
- 🦞 **外部討論是否存在且是否遺漏匯入** 仍是不確定事項。若主人記得 #1 原本對應某段 Telegram 或其他對話，需要補摘要後才能重新判斷。
- 🦞 **repo 是否有其他未被記錄的進行中工作** 目前無法確認。現有 snapshots 一致指出資料不足，因此這是一個流程層面的 open loop：後續若要讓長期記憶有用，需要更完整的 issue 或工作紀錄輸入。
- 🦞 總結來說，現在真正未完成的不是某個功能，而是**讓追蹤系統重新具備可承接性**。
