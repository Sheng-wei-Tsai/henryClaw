# Repository Memory

## Stable Context
- 龍蝦群目前能穩定確認的 repo 脈絡非常少；近一段時間內，**唯一持續可見的工作入口是 issue `#1`**。
- `#1` 長期維持 **open**，且已指派給 **主人（Sheng-wei-Tsai）與 Copilot**。這表示 repo 至少存在一個預期由人機協作承接的工作容器。
- 這個 repo 目前呈現的工作模式，比較像是 **先建立追蹤入口，再等待後續補齊內容**，而不是已經有明確需求、拆解與實作節奏的專案。
- 龍蝦群對記憶整理採取保守原則：**只記錄已知事實，不替主人腦補需求、優先順序或技術方案**。這點在多日快照中反覆一致，可視為目前的穩定整理準則。
- 目前**看不出明確產品方向、功能主題、模組邊界或技術決策**；不是因為已確認不存在，而是因為可用資訊不足，尚不能安全下結論。

## Recent Themes
- 最近反覆浮出的主題不是功能開發，而是 **資訊缺口管理**。龍蝦們每天巡邏時，看到的重點都在於：有 issue 容器，但內容尚未成形。
- `#1` 一直被視為 **未定義完成的記憶支線／工作掛點**：
  - 有追蹤位置
  - 有指派對象
  - 但缺少任務目標、背景、驗收條件與討論脈絡
- **協作意圖存在，但協作契約未落地**：主人與 Copilot 都已被掛上 `#1`，顯示原本期待協作；然而目前沒有足夠資訊說明誰該做什麼、做到哪裡算完成。

- 整體來看，近期主題可濃縮為一句：**這個 repo 目前先有殼，還沒有肉；先有掛點，還沒有可執行任務。**

## Constraints
- **資訊不足是目前最大的限制。** 除了 `#1` 之外，沒有足夠 issue 網絡可供交叉驗證，也沒有留言、標籤、需求描述或驗收標準可供推進。
- **不能把空白 issue 當成已啟動工作。** `#1` 雖然 open，但目前更像是占位容器，而非可直接執行的任務。
- **不得臆測需求內容。** 在缺少標題語意、內文、背景對話與完成條件時，龍蝦群應維持保守，不虛構產品目的、技術方向或優先順序。
- **外部對話未回填會削弱工程可讀性。** 若需求或決策只存在於 Telegram 等 repo 外來源，後續 agent 很難延續上下文。
- **目前沒有可確認的長期決策或穩定規則，除了保守記憶治理本身。** 若未來出現新資訊，這份記憶需要重新校正。
- **存在不確定性：** `#1` 究竟是生活待辦、工作任務、產品功能、研究草稿，還是單純測試用 issue，目前無法判定。
