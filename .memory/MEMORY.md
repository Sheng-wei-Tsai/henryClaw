# Repository Memory

## Stable Context
- 龍蝦群目前能穩定確認的 repo 脈絡非常薄：近一段時間內，唯一反覆出現且可辨識的 issue 是 **#1**，它已被判定為**沒有形成實質需求的占位／測試性卡片**，且已關閉。
- 這個 repo 目前**看不出穩定的產品主線、功能開發節奏或長期工程分支**。不是因為一定沒有工作，而是因為現有 issue 記錄不足以支撐這種判讀。
- 可長期保留的治理共識是：
  - **沒有明確目標、背景、討論或驗收條件的 issue，不應視為有效工作入口。**
  - **已確認為空白或占位用途的 issue，應直接關閉或歸檔，避免污染 backlog。**
  - 若未來要重啟已失去上下文的方向，**應重新開新 issue**，而不是依賴舊的空白卡片續做。
- 手動筆記本身沒有額外的人類長期規則；目前長期記憶主要來自 daily snapshots 的一致訊號。

## Recent Themes
- 近期最明顯的主題不是功能推進，而是**任務池清理與資訊衛生**。龍蝦們反覆確認：
  - `#1` 不承載實際需求。
  - `#1` 不值得再投入追查成本。
  - 關閉這類空白卡片，比保留虛假進度更有價值。
- 另一個反覆出現的近期主題是**記錄品質不足**：
  - 缺少需求描述。
  - 缺少背景脈絡。
  - 缺少討論留言或外部對話摘要。
  - 缺少驗收標準。
- 因此，近期 repo 呈現出的不是「多線開發」，而是「**先把不能信任的待辦清掉**」。這代表主人或協作者至少在做一件對長期維護有幫助的事：讓 backlog 更乾淨、更可信。
- 需要明說的不確定性：目前 snapshots 幾乎都只圍繞 `#1`，所以**無法可靠推論主人近期是否其實有其他未 issue 化的工作、生活流程或工程進展**。

## Constraints
- **資料來源限制很強**：目前可用記憶幾乎只來自 issue 摘要；若工作沒有進入 issue，龍蝦群就很難替主人保留可接手的長期脈絡。
- **禁止腦補需求**：當 issue 缺少目標、背景與討論時，龍蝦群的共識是**不自行延伸推論內容**，避免把空白殼誤記成真任務。
- **可接手性依賴記錄品質**：若未來希望 daily memory snapshots 有實際價值，issue 至少應補上：
  - 任務目的
  - 背景／來源脈絡
  - 預期成果或驗收條件
  - 若有外部討論，附上連結或摘要
- **目前沒有足夠證據建立更細的 repo 慣例**，例如技術棧、部署流程、協作節奏、標籤規則等；若硬寫會超出資料支持範圍。

## Open Loops
- **`#1` 的原始意圖仍不明。** 若主人其實還記得當初想做什麼，現在的記錄已不足以延續，應重新建立新 issue 並補齊上下文。
- **是否存在未被記錄的其他工作線仍未知。** 目前 snapshots 無法判斷主人是否有其他進行中的工程、生活安排或協作事項，只能誠實標記為資料不足。
- 若主人未來想讓這個 repo 的長期記憶更有用，待補的不是 `#1` 本身，而是**新的、可執行且有上下文的任務紀錄方式**。
- 目前**沒有可確認的技術 blocker、功能缺陷或已展開但未完成的工程項目**；這是因為資料不足，不代表一切都已完成。
