# Repository Memory

## Stable Context
- 龍蝦群目前能穩定確認的 repo 脈絡非常少；近 30 天可見的 issue 幾乎只有 **#1**，且它已於 **2026-05-15** 關閉。
- **#1 已被反覆確認為占位／測試性紀錄，而非真正展開過的工程任務。** 它缺少需求描述、背景脈絡、討論紀錄、驗收條件與實作進度，因此不應被視為有效待辦。
- 這個 repo 目前可提煉出的長期工作記憶，不是某個產品方向或技術主線，而是較偏向**記憶治理原則**：  
  - 沒有明確需求的 issue，不應自動延伸成任務。  
  - 關閉且內容空白的 issue，應視為已收束的噪音來源，而不是潛在 backlog。  
  - 若未來要重啟空白議題，應補齊目標、範圍與預期成果，必要時直接開新 issue。
- 目前**沒有足夠證據**可建立更高層的穩定 repo context，例如產品目標、技術棧偏好、部署流程、協作節奏或生活安排模式；這些都應誠實標記為未知，而不是腦補。

## Recent Themes
- 最近幾天反覆出現的主題是**記憶清點與去雜訊**，不是功能開發或 bug 修復。龍蝦們主要在替主人確認：哪些東西其實沒有內容，不值得繼續追。
- **Issue hygiene / 任務衛生** 是近期最明顯的工作訊號：空白 issue 不應長期漂浮在工作視野中，避免造成「好像還有事沒做」的假壓力。
- **需求明確性** 一再被凸顯：若 issue 沒有「為什麼要做、要解決什麼、完成長什麼樣子」，龍蝦群就無法可靠接手，也不應替主人補寫不存在的規格。
- 多份 daily snapshot 都指出**資料密度不足**：目前看不出其他穩定主題、跨 issue 關聯、技術債、產品決策或生活工作節奏。這代表近期記憶更像是「確認空白」，而不是「累積新脈絡」。

## Constraints
- **不可腦補需求。** 當 issue 缺乏上下文、留言或外部對話證據時，龍蝦群應採保守整理，不虛構任務拆解、排程或技術結論。
- **GitHub issue/comment 才是主要原始來源。** 若真正脈絡散落在 Telegram 或其他 repo 外工具，沒有補回摘要前，記憶系統無法可靠重建上下文。
- **空白 issue 不等於待辦。** 缺少目標、範圍、驗收條件的項目，不應被當成有效工作入口。
- **目前資訊不足是實際限制。** 由於可見資料幾乎只有 #1，任何關於 repo 長期方向、主人近期實作內容或協作模式的推論都高度不確定。
- 若未來希望 daily memory 更有用，至少需要在 issue 中留下：
  - 需求背景
  - 任務目標
  - 決策理由
  - 預期成果或驗收方式

## Open Loops
- **若主人想重啟 #1**：需要先補上明確需求、背景、範圍與預期輸出；否則這條線仍應視為已結束。
- **是否存在 repo 外遺失脈絡**：多份快照都提到 Telegram 或其他外部討論可能缺席；目前無法確認 #1 或其他工作是否曾在外部被討論過。
- **Issue 建立流程是否需要更標準化**：目前只看到一個空白 issue 案例，證據有限，但它已暴露出「有票但無內容」會讓後續記憶整理失效。這是值得留意、但尚未能定義為既定規則的主題。
- **是否有其他未 issue 化工作正在進行**：現有資料看不出來。龍蝦們只能記住這個不確定性，不能把沉默誤判成沒有工作。
