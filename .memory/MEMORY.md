# Repository Memory

## Stable Context
- 🦞 這個 repo 目前可確認的長期脈絡非常薄，近一段時間內**沒有穩定可辨識的產品主線、功能開發節奏或多 issue 協作模式**。
- 🦞 龍蝦群已形成一個可重用的判讀原則：**沒有需求描述、背景、驗收條件與討論脈絡的 issue，不應被當成可執行工作**。這類條目比較適合視為占位、測試或未成形紀錄。
- 🦞 目前唯一反覆被提及的 issue 是 **#1**，且已被一致判定為：
  - 無實質需求內容
  - 缺乏留言與可追溯背景
  - 已於 **2026-05-15** 關閉
  - 不應再視為進行中任務
- 🦞 因此，對這個 repo 來說，現階段較穩定的「上下文」不是某個功能，而是**記憶整理優先於虛構脈絡**：寧可承認資料不足，也不要替主人腦補不存在的工作故事。
- 🦞 `shared/manual.md` 目前沒有額外人工補充的長期規則；可依其說明理解為：**GitHub issue/comment 才是原始資料來源，手動筆記應只保留蒸餾後的長期記憶**。

## Recent Themes
- 🦞 近期最明顯的主題不是開發，而是**議題衛生與追蹤品質管理**。龍蝦們反覆確認：空白或未成形的 issue 不該長期佔據注意力。
- 🦞 **#1 的關閉**被視為一種清理動作，而不是功能完成。重點在於把「沒有內容的殼」正式歸檔，避免未來誤判成漏追待辦。
- 🦞 近期 daily snapshots 一再顯示：**資料量極少，且沒有其他可串聯的 issue 主題**。因此看不出明確的技術方向、產品決策、除錯脈絡或生活事件與 repo 工作的連動。
- 🦞 另一個反覆出現的訊號是：**若外部對話（例如 Telegram）沒有被匯入或對齊，記憶鏈就會斷裂**。這不是已證實的流程問題，但已多次被提到，值得當成近期觀察主題，而非穩定事實。

## Constraints
- 🦞 **資訊不足必須明講**：當 issue 缺少內容時，不能從空白標題、空白內文或關閉狀態硬推導需求。
- 🦞 **GitHub issue/comment 是主要可依據來源**；若真正脈絡存在於 Telegram 或其他外部管道，但未被整理進 repo 記憶，龍蝦群就無法可靠接手。
- 🦞 **關閉不等於有交付成果**：像 #1 這類案例，關閉只代表停止追蹤，不代表功能完成。
- 🦞 目前**沒有足夠證據**支持以下任何結論：
  - repo 正在推進某個穩定專案主線
  - 主人近期有明確工程里程碑
  - 存在可辨識的標籤策略、分工慣例或技術決策模式
- 🦞 若未來要讓記憶系統更可靠，issue 至少需要具備：
  - 目標
  - 背景脈絡
  - 預期產出或驗收條件

## Open Loops
- 🦞 **#1 是否曾對應某個真實需求**，目前仍無法完全排除；只是現有資料不足，不能把它當成已知任務。
- 🦞 若主人未來想重啟 **#1** 或沿用其方向，需先補齊：
  - 目標是什麼
  - 為何要做
  - 預期成果／驗收條件是什麼
  - 是否有外部對話或背景需要一併補回
- 🦞 目前看不到其他明確進行中的工程 open loop；但這個結論的前提是：**現有記錄本身非常稀薄**。若主人其實有在 repo 外推進工作，龍蝦群現在還沒抓到那條水流。
- 🦞 一個較大的未解環是：**這個 repo 未來是否會承載真正的工作流，還是目前仍處於測試／佔位階段**。現有資料不足以下定論。
