# Repository Memory

## Stable Context
- 🦞 這個 repository 目前**幾乎沒有可確認的長期工程脈絡**；近多日可見的 issue 記憶中，唯一反覆出現的是 **#1**，而它已被一致判定為**空白占位／測試性質的 issue**，不是實際展開過的任務。
- 🦞 龍蝦群已形成穩定共識：**已關閉但缺乏目標、背景、討論與驗收資訊的 issue，不應被視為可延續的工作依據**。像 #1 這類卡片，應視為記憶噪音已清理，而不是待辦。
- 🦞 目前可保留的 repo 層級習慣是偏保守的記憶治理原則：
  - 不替空白 issue 腦補需求。
  - 不因為卡片存在就假設有實際工作。
  - 若要讓 issue 成為可接手的記憶單位，至少需要留下**任務目標、背景脈絡、預期成果／驗收條件**。
- 🦞 共享手動筆記目前沒有額外的人類長期規則；因此這份記憶主要來自 daily snapshots 的反覆訊號，而那些訊號本身也顯示：**現階段 repo 的長期上下文仍非常稀薄**。

## Recent Themes
- 🦞 最近這段時間的主題不是功能開發，而是**任務池清理與記憶衛生**。龍蝦群反覆在做的事，是確認哪些 issue 其實沒有內容、哪些不值得再追查。
- 🦞 **#1 已被收束為無效工作項**：它沒有需求描述、沒有留言討論、沒有實作紀錄，也沒有可接回生活／工作情境的外部脈絡；因此近期整理重點是避免主人未來再把它誤認為活躍任務。
- 🦞 **資訊完整性不足** 是最近最明顯的反覆訊號。多日摘要都指出：若 issue 沒有補齊背景與目標，龍蝦群就無法可靠接手，也無法從 issue 還原主人真正的工作進展。
- 🦞 另一個近期主題是**資料不足本身**。從 snapshots 看不出多條功能線、技術依賴、產品方向或生活安排的穩定模式；這不一定代表主人沒有在做事，更可能代表**工作沒有被充分記錄進 issue 記憶**。

## Constraints
- 🦞 **資料來源限制很強**：目前近 30 天可整理出的 issue 幾乎只有 #1，且內容空白，因此很多結論只能停留在「無法確認」，不能延伸推論。
- 🦞 **不可憑空補故事** 是這個 repo 的重要限制。當 issue 缺少需求、背景、討論串或外部對話摘要時，龍蝦群應保守處理，不把它寫成已知事實。
- 🦞 **關閉狀態不等於完成了某個功能**；在這個 repo 的現有脈絡裡，關閉更可能只是代表「這張卡片不值得繼續追蹤」。
- 🦞 若未來主人透過 Telegram 或其他外部對話形成需求，但**沒有把摘要或連結帶回 issue**，那麼 repo 記憶會斷鏈，後續 agents 幾乎無法接手。
- 🦞 目前沒有足夠證據建立穩定的技術棧、協作流程、產品方向或開發節奏；這些都應視為**未知**，而不是默認存在。

## Open Loops
- 🦞 **若主人想重啟 #1 對應的方向，必須重新定義**。目前缺少的核心資訊包括：
  - 任務目標
  - 背景脈絡
  - 預期成果或驗收條件
  - 若有外部討論，需補上連結或摘要
- 🦞 **是否存在未被 issue 記錄的其他工作脈絡**，目前無法判斷。多日 snapshots 都指出資料可能嚴重不足，這是尚未解開的記憶缺口。
- 🦞 **未來 issue 建立品質** 仍是待改善的流程議題：如果主人希望龍蝦群能持續整理出有用的長期記憶，後續新 issue 需要更完整的上下文，而不是只留下空殼。
- 🦞 不確定性聲明：目前沒有任何可可靠列為「正在進行中的工程任務」；這可能代表真的沒有，也可能只是記錄缺漏。龍蝦群暫時只能把這件事記成**未確認**。
