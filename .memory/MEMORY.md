# Repository Memory

## Stable Context
- 龍蝦群目前能確認的 repo 長期脈絡非常薄，近 30 天幾乎只看到一張可辨識的 issue：**#1**。
- **#1 已於 2026-05-15 關閉**，且長期被反覆判定為：
  - 占位／測試性節點
  - 沒有明確需求
  - 沒有討論脈絡
  - 沒有驗收條件
  - 不應視為可執行任務
- 這個 repo 目前可穩定記住的工作習慣是：**對資訊不足的 issue 採保守處理**。龍蝦們不會替主人腦補需求，也不會把空白票據延伸成虛構的工程工作。
- 另一個可蒸餾出的穩定規則是：**issue hygiene 很重要**。沒有內容承載力的 issue，應被辨識、關閉或停止追蹤，避免 backlog 與長期記憶被雜訊污染。
- 目前**看不出穩定的產品方向、技術主線、開發節奏或跨 issue 關聯**；資料量不足，不能假裝 repo 已有成熟工程敘事。

## Recent Themes
- 最近幾天的主旋律不是功能開發，而是**記憶清點與任務去雜訊**。
- 龍蝦們反覆確認同一件事：**#1 不構成真正工作主線**，其價值主要在於提醒主人「這裡沒有東西可追」。
- 近期反覆浮現的訊號包括：
  - **需求明確性不足會直接阻斷後續協作**
  - **空白 issue 會增加認知負擔**
  - **若工作脈絡散落在 repo 外（例如聊天或 Telegram），回頭整理時會難以重建上下文**
- 整體來看，主人最近在這個 repo 的節奏比較像：
  - 清理模糊入口
  - 回收注意力
  - 保留最小必要記錄
  - 暫時沒有可確認的新功能、除錯或文件主題在穩定推進
- 不確定性：這不代表主人真的沒有做事，只代表**目前 issue 記憶不足以還原其他工作面向**。

## Constraints
- **資料來源稀薄**：近 30 天可用 issue 幾乎只有 #1，無法可靠推導更大的專案脈絡。
- **禁止腦補**：當 issue 缺少背景、目標、討論與驗收條件時，龍蝦群應維持保守，不自行補完需求。
- **空白 issue 不算有效待辦**：沒有功能目標、範圍與預期成果的票據，不應進入實作、排程或技術拆解。
- **外部脈絡缺失會削弱記憶品質**：若真正的需求討論發生在 Telegram 或其他 repo 外通道，但沒有回填摘要，後續 agents 幾乎無法接手。
- **目前無法建立穩定 repo 規範全貌**：除了「需求要清楚、空白票據不要追」之外，尚無足夠證據整理出更多長期工程慣例。

## Open Loops
- **若 #1 要復活，主人需要重新補件**：
  - 為什麼要做
  - 想解決什麼問題
  - 範圍是什麼
  - 預期成果／驗收條件是什麼
- **是否存在 repo 外遺失脈絡**：若 #1 或其他工作曾在 Telegram、聊天紀錄或其他地方討論，仍待補回摘要或連結。
- **是否需要建立 issue 描述模板**：近期反覆暴露出「有 issue 但沒有任務內容」的問題，值得考慮用模板強化需求背景、目標與驗收標準。
- **是否其實有其他工作未被 issue 化**：目前看不出主人近期真正推進中的主線；這可能代表工作不存在，也可能只是沒有被記錄進這套記憶系統。
- **repo 的長期方向仍不明**：在更多有效 issue 或人工筆記出現前，龍蝦群無法可靠判定這個 repository 的產品目標、技術重心或協作模式。
