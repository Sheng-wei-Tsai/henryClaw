# Repository Memory

## Stable Context
- 龍蝦群目前能穩定確認的 repo 脈絡很少；`Sheng-wei-Tsai/henryClaw` 在這段觀測期間幾乎沒有可執行的 issue 內容，主要只看得到一個開啟中的 **#1** 作為記憶入口。
- **#1 長期呈現為占位／掛點型 issue**：它比較像協作與記憶承接的容器，而不是已定義清楚的任務單。
- 主人與協作代理的工作方式，傾向於把後續脈絡整理成**可持續接手的濃縮記憶**，而不是保存逐字聊天紀錄。這是目前少數可辨識、且跨多日重複出現的穩定方向。
- 目前沒有足夠證據支持任何產品主線、技術棧選型、功能優先序或固定開發流程；若要記住這個 repo 的現況，最準確的說法是：**記憶基礎設施先於實作內容**。
- 不確定性很高：若主人在觀測期間有其他工作進展，現有 snapshots 並未留下足夠痕跡，龍蝦群不能可靠重建。

## Recent Themes
- **資訊不足是主要阻塞**：近幾日反覆出現的核心問題不是技術難題，而是缺少任務目標、背景、預期成果與成功條件。
- **先保留入口、等待內容補齊**：龍蝦群多次確認 #1 應先作為後續工作的掛點保留，而不是在資訊不足時硬拆任務或預設方向。
- **工程可讀性／可交接性被重視**：從多日摘要可看出，主人似乎在意後續是否能讓代理或未來的自己順利接手，因此偏好結構化、濃縮過的記憶。
- **近期活動趨近停滯**：4/25–4/27 的快照都顯示沒有新的可用 issue；近期主題因此從「辨識資訊缺口」進一步轉為「等待下一輪更新」。

## Constraints
- **不可臆測需求**：在 #1 缺少明確描述時，龍蝦群共同遵守的限制是，不自行定義工作範圍、不推論優先級、不假設技術方案。
- **目前只能依賴 issue / comment 作為原始來源**：若沒有留言、對話紀錄或補充背景，就無法建立可靠的決策歷史。
- **缺乏可追溯上下文**：目前看不到足夠的 Telegram 對話、需求澄清或驗收條件，因此記憶接續能力受限。
- **資料稀薄，無法估工或排程**：在任務目標、完成條件與影響範圍未定前，不適合做實作拆解、時程安排或狀態判定。
- **觀測面有限**：近幾日甚至沒有可用 issue；因此任何「repo 很安靜」的結論，都應理解為「在目前可見資料中很安靜」，不是對主人整體工作狀態的全面判定。
