  - 先保留線索；
  - 再等待需求與脈絡補齊；
  - 不因空白 issue 而自行擴張解讀。

  近幾天反覆出現的不是功能開發、修 bug 或規格收斂，而是龍蝦群持續確認：#1 缺少標題語意、內文、留言、外部對話與完成條件，因此暫時不能推進。
- **主題二：協作入口已存在，但任務語意尚未落地。**  
  issue 已建立、也有指派痕跡，表示主人可能先把入口留好；但這條入口還沒被餵進足夠上下文，所以目前只能當作待補資料的節點。
- **主題三：記憶治理優先於貿然行動。**  
  龍蝦群近期反覆採取同一策略：不猜需求、不拆任務、不排優先級。這不是停滯，而是在資訊密度過低時，優先保護記憶品質與可追溯性。
- **主題四：可追溯來源不足。**  
  多日摘要都提到缺少 issue 內文、留言與 Telegram 對話脈絡。這代表目前最大的阻塞不是技術，而是**需求來源沒有被整理進 repo 可讀記憶**。
- **主題五：近期沒有可確認的實作進展。**  
  在現有資料裡，看不出近期有明確完成的工程、文件、流程優化或生活支線整理成果；若主人實際上有推進其他事情，這批 snapshots 沒有足夠證據反映出來。

## Constraints
- **不可臆測需求。**  
  在缺少標題、內文、留言或外部對話時，龍蝦群不應自行定義 #1 的目標、範圍、優先順序或解法。
- **Issue / comment 才是原始資料來源。**  
  手動記憶與 compact memory 只應做蒸餾，不應把空白處補成看似確定的事實。
- **目前資料可讀性極低。**  
  #1 缺少足夠上下文，導致無法可靠估工、排程、拆解、驗證完成條件，也無法判斷是否影響其他工作。
- **跨議題關聯幾乎不存在。**  
  近期只有 #1 可見，沒有其他 issue 可互相佐證，因此任何「repo 正在做什麼」的結論都必須保守。
- **不確定性需要明講。**  
  目前無法確認：
  - #1 是工程任務、文件整理、流程優化、生活提醒，還是單純占位；
  - 主人是否有其他未被匯入的工作脈絡；
  - 人機協作在此 repo 中是常態，還是只出現在這一條支線。
