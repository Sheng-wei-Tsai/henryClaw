# Repository Memory

## Stable Context
- 龍蝦群目前能確認的穩定脈絡很少：這個 repo 的記憶整理流程依賴 **GitHub issue / comment** 作為原始事實來源，記憶文件應該是整理後的長期脈絡，而不是逐字備份原始討論。
- `.memory/shared/manual.md` 是給人類手動維護的長期記憶區，適合放：
  - 穩定規則
  - 長期決策
  - 常見限制
  - agent 共同遵守的 repo 習慣
- compact-memory workflow 會讀取手動筆記，但**不會覆寫**它；因此手動筆記可視為較穩定的人類維護層。
- 目前從近幾日 snapshots 看不出任何可穩定歸納的產品、技術棧、工作流程或主人偏好；這部分資訊仍不足。

## Recent Themes
- 最近一整段整理視窗（2026-06-14 至 2026-06-20）**都沒有可用 issue**，因此沒有新的工作主題浮現。
- 每日摘要反覆指出：
  - 沒有 agent 活動可供彙整
  - 沒有跨 issue 主題
  - 沒有新的跨 issue 決策
- 換句話說，近期主題不是「某項功能正在推進」，而是 **記憶來源暫時空窗**；龍蝦們目前只能守住既有框架，等待新訊號進來。

## Constraints
- 記憶整理必須以 **長期可重用的 repo context** 為目標，避免把 daily snapshots 或 issue 內容整段搬進來。
- 若某件事只短暫出現、尚未證明是穩定事實，應放在近期主題或 open loop，而**不應寫成長期規則**。
- 目前最大的限制是：**資料不足**。近七天沒有 issue 可供交叉比對，因此無法可靠推定：
  - 主人的近期工作重心
  - repo 的活躍模組
  - 穩定的工程慣例
  - 未完成工作的優先順序
- 目前資訊也不存在明顯衝突；問題不是矛盾，而是**缺少可驗證內容**。

## Open Loops
- 等待下一輪有實際 issue / comment 更新後，再重新整理長期記憶。
- 需要未來資料來確認：
  - 這個 repo 的核心用途與主要維護方向
  - 是否存在固定的開發／回覆／整理習慣
  - 是否有主人反覆強調的限制、偏好或決策準則
- 若人類已知有穩定規則但尚未出現在 issue 中，建議直接補進 `.memory/shared/manual.md`，讓龍蝦群之後能更穩地接手記憶整理。
