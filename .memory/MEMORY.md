# Repository Memory

## Stable Context
- 龍蝦群目前能穩定確認的 repo 脈絡非常少；近一段時間內，**沒有可驗證的持續開發主線、產品方向或技術決策** 被留下來。
- 目前唯一反覆出現且可視為穩定結論的事項是：**Issue #1 已被判定為空白占位／測試性條目，且已於 2026-05-15 關閉**，不應再被視為有效待辦或工作入口。
- 這個 repo 的記憶整理原則偏保守：**沒有明確需求、背景、討論或驗收條件，就不把它當成可承接的工程任務**。
- 從多日 snapshot 一致可見，龍蝦群會把 issue 當作重要記憶來源，但也明確知道：**若 issue 本身內容不足，它就不能支撐長期記憶**。

## Recent Themes
- 最近反覆出現的主題不是功能開發，而是**清理追蹤系統中的雜訊**：把沒有內容的 issue 明確歸檔，避免誤導後續注意力。
- 多日摘要都指向同一件事：**資訊不足比技術阻塞更明顯**。目前看不到卡在某個 bug、架構選型或產品規格，而是根本缺少足夠上下文來判斷下一步。
- 龍蝦們持續提醒一個近期訊號：**記錄完整性不足會直接中斷協作與記憶延續**。若需求只存在於外部對話、Telegram 或腦中，卻沒有回填到 issue，後續就很難接手。
- 整體工作節奏在這批資料裡偏靜態；近期比較像是**確認哪些事情其實沒有開始**，而不是推進新的實作航線。

## Constraints
- **資料來源限制明顯**：目前 memory 幾乎只反映到一張內容不足的 issue，無法可靠推導 repo 的真實用途、技術棧、開發流程或生活工作節奏。
- **不可憑空補完需求**：對空白 issue 或缺乏上下文的條目，必須維持保守，不推測目標、不虛構進展、不假設已完成的功能。
- **Issue 品質直接限制記憶品質**：若沒有背景、目標、範圍、驗收條件與討論紀錄，龍蝦群就只能把它標記為不可承接的殼。
- **外部對話可能遺失**：多份 snapshot 都提到 Telegram 或其他外部脈絡可能未匯入；這代表目前記憶可能不完整，需明確承認不確定性。
- **目前沒有足夠證據建立穩定 repo 規則**，除了「空白 issue 不應視為有效任務」之外，其餘流程習慣仍屬未知。

## Open Loops
- **若 #1 背後其實有真實需求，該需求仍未被重新定義**。目前缺少：
  - 目標
  - 背景脈絡
  - 範圍界定
  - 驗收條件
- **是否存在未匯入的外部討論** 仍不確定；若有 Telegram 或其他來源的關鍵對話，需補回來才能判斷 repo 是否其實有被中斷的工作線。
- **未來若要重啟 #1 相關想法**，較合理的做法是：
  - 另開新 issue；或
  - 先補齊完整上下文後再重新啟動  
  而不是直接沿用目前這張空白殼。
- **repo 的實際長期用途仍不明**：現有資料不足以判定這裡主要承載產品開發、個人工具、流程實驗，或只是測試倉庫。這點目前必須保留不確定性。
