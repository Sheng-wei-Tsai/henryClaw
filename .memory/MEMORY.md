# Repository Memory

## Stable Context
- 龍蝦群目前能穩定確認的長期脈絡很少；這個 repo 在最近可見的記憶視窗內，**幾乎沒有持續活躍的 issue 流**，因此尚未形成清楚的產品主線、技術慣例或工作節奏。
- 目前唯一曾被明確提到的工作入口是 **Issue #1**。它比較像是後續工作的**掛點／記憶錨點**，而不是已經定義完整的任務。
- 從共享手動筆記可知，這份記憶系統的定位很明確：
  - **GitHub issue / comment 是原始事實來源**
  - `shared/manual.md` 用來保存**人類手動維護的長期記憶**
  - compact-memory workflow 會讀取手動筆記，但**不會覆寫**它
- 因此，龍蝦群在整理長期記憶時，應優先做**蒸餾與歸納**，而不是複製 issue 原文或把 daily snapshot 當成永久事實。

## Recent Themes
- 最近一段時間的主題其實很單純：**資訊稀少、工作面安靜、等待更多上下文**。
- 在 2026-04-24 的快照中，龍蝦群曾確認：
  - **#1 已存在且已指派給主人與 Copilot**
  - 但它仍缺少足夠內容，無法轉成可執行任務
  - 當時主要進展不是實作，而是**辨識資訊缺口**
- 之後從 2026-04-25 到 2026-04-30，連續多天都沒有新的可用 issue 資訊；可視為近期反覆出現的模式：
  - 沒有新的跨 issue 主題
  - 沒有新的跨 issue 決策
  - 記憶整理工作暫時停留在「保留既有脈絡、等待更新」
- 綜合來看，近期主題不是某個功能或專案，而是：
  - **repo 目前缺乏足夠任務描述**
  - **記憶延續依賴未來 issue 更新**
  - **尚未觀察到穩定的開發／整理主線**

## Constraints
- **原始資料不足**：最近多日沒有可用 issue 內容，導致龍蝦群無法可靠推斷需求、優先順序或技術方向。
- **#1 內容不足**：已知它是入口，但缺少目標、背景、預期成果等核心資訊，因此不能把它當成已定義完成的工作規格。
- **不可把短暫訊號誤寫成穩定規則**：目前很多觀察只來自單一日期或單一 issue，穩定性不足。
- **記憶來源有層級**：
  - issue / comment 才是原始來源
  - 手動筆記是長期規則補充
  - daily snapshots 是整理結果，不應直接視為永久真相
- **存在不確定性**：目前無法確認這個 repo 的主要用途、技術棧、協作流程或主人近期真正想推進的事項；若未來資料出現，這些判斷都可能需要更新。

## Open Loops
- **Issue #1 仍待補完**：
  - 目標是什麼
  - 背景脈絡是什麼
  - 預期成果／驗收條件是什麼
- **角色分工未明**：雖然 #1 曾指派給主人與 Copilot，但尚不清楚各自負責開發、整理、追蹤，還是僅作為記憶掛點。
- **缺少可延續的決策歷史**：目前看不到足夠的對話、補充說明或後續更新，龍蝦群只能先守住入口，無法穩定接續更深的工作記憶。
- **等待下一輪 issue 活動**：在沒有新資料前，龍蝦群不應虛構產品方向或技術決策；目前最合理的待辦，就是等主人或協作者補上更多可驗證資訊。
