# Repository Memory

## Stable Context
- 龍蝦群目前能確認的長期脈絡非常少；這個 repo 在近一段時間內，**幾乎沒有可追蹤的實質 issue 工作流**。
- 唯一反覆被確認的議題是 **`#1` 已於 2026-05-15 關閉**，且內容近乎空白，較像**占位／測試用 issue**，不是可延伸的正式任務主線。
- 對 `#1` 的穩定判讀是：
  - 不承載明確需求
  - 沒有足夠背景、討論或驗收資訊
  - 不應作為後續功能、規格或待辦的依據
- 這個 repo 目前更像是在建立／清理記憶治理基礎，而不是已有穩定的產品開發節奏。龍蝦們要替主人記住：**不要把空白節點誤認成正在推進的工作**。

## Recent Themes
- 最近幾天反覆出現的主題不是功能開發，而是**記憶清理與 issue 衛生**：
  - 辨識哪些 issue 其實沒有內容
  - 接受其結案狀態
  - 避免未來重複花時間回頭翻找空殼
- 龍蝦群多次觀察到：這段時間的主要價值在於**降低追蹤噪音**，而不是擴張任務面。
- 另一個反覆浮現的訊號是**資料密度不足**：
  - 看不到明確需求拆解
  - 看不到決策討論
  - 看不到實作、除錯或驗收脈絡
  - 也無法確認是否有 repo 外（例如 Telegram）上下文遺失
- 因此，近期主題可總結為：**先把沒有內容的支線收掉，等待未來以更完整的方式重新開案**。

## Constraints
- **資訊不足是目前最大的限制。** 不能從現有 issue 記錄可靠推導產品方向、技術債、生活安排或跨任務依賴。
- 龍蝦群應遵守保守原則：**沒有需求與上下文，就不要腦補任務、規格或背景故事**。
- 目前可見記錄顯示，repo 內 issue 可能不足以承載完整工作記憶；若主人有在其他地方討論（如 Telegram），這些脈絡尚未回填。
- 若未來要讓記憶系統可用，issue 至少需要具備：
  - 背景
  - 目標
  - 範圍
  - 預期成果或驗收條件
- 目前沒有足夠證據證明這些已成為正式 repo 規範；只能說這是從近期反覆缺口中推得的**強烈建議**，不是已落地的制度。

## Open Loops
- **是否需要建立更一致的 issue 模板？** 近期反覆暴露出「有 issue 但沒有任務內容」的問題，但尚未看到正式決議或實作。
- **是否存在 repo 外遺失的上下文？** 多份 snapshot 都提到可能缺少 Telegram 或其他外部討論；目前無法確認。
- **主人是否其實有其他進行中的工作，但尚未 issue 化？** 現有資料看不出來，這是明確的不確定性。
- **若未來要重啟 `#1` 背後的想法**，應視為新工作重新定義，而不是依附在舊殼上；但目前沒有任何足夠資訊說明那個想法究竟是什麼。
- 除此之外，**沒有可直接從 `#1` 延伸的實質待辦**；這條支線可暫時視為已完全收束。
